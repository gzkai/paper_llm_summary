## 研究背景与动机

### 核心洞察

每次智能体执行动作 $`a_t`$ 后，环境都会返回一个 **next-state signal** $`s_{t+1}`$（用户回复、工具执行结果、GUI 状态变化、测试结果等）。现有系统仅将 $`s_{t+1}`$ 作为下一步的上下文，却忽视了其中蕴含的两类宝贵学习信号：

### 两类被浪费的信号

**Waste 1 — 评估信号（Evaluative Signals）**：next-state signal 隐式地对上一个动作打分：
- 用户重新提问 → 不满意
- 测试通过 → 成功
- 错误日志 → 失败

这构成了天然的过程奖励（Process Reward），无需额外标注，但现有系统要么忽视它，要么只在离线预收集的数据上利用它。

**Waste 2 — 指令信号（Directive Signals）**：next-state signal 往往不只是评分，还携带**如何改正**的方向信息：
- 用户说"你应该先检查文件再编辑" → 不仅知道回答错了，还知道**哪些 token 应该不同**
- SWE 错误追踪 → 隐含具体的修正方向

标量奖励（scalar reward）完全丢失了这种方向性信息。

### 现有方法的不足

| 方法类别 | 局限 |
|---------|------|
| RLVR（GRPO/PPO） | 使用标量奖励，无法将指令信号转化为方向性梯度 |
| DPO | 需要成对偏好数据，依赖预收集数据集 |
| 蒸馏方法 | 依赖预先整理的反馈-回复对，而非实时信号 |
| 现有 Agentic RL | 各自针对单一环境，无统一框架；均为离线批量训练 |

## 核心方法：OpenClaw-RL

OpenClaw-RL 是一个**统一框架**，同时支持：
- **个人智能体（Personal Agents）**：从用户对话中持续个性化学习
- **通用智能体（General Agents）**：Terminal、GUI、SWE、Tool-call 等大规模 RL 训练

### 1. 异步四解耦架构

```
Policy Serving → Environment → Reward Judging → Policy Training
  (SGLang)       (Http/API)    (SGLang/API)      (Megatron)
```

四个组件完全解耦、异步运行，**互不阻塞**：
- 模型服务下一个请求的同时，PRM 在评判上一个回复，训练器在应用梯度更新
- 对个人智能体：通过保密 API 连接用户设备，无需修改智能体框架，权重更新不中断推理
- 对通用智能体：云端托管环境支持大规模并行（Terminal: 128 并行，GUI/SWE: 64，Tool-call: 32）

### 2. 会话感知环境服务器

将每个 API 请求分类为：
- **主线 turn（Main-line turn）**：智能体主要回复和工具执行结果 → 产生训练数据
- **支线 turn（Side turn）**：辅助查询、记忆整理等 → 不产生训练数据

每个新主线请求的消息包含上一轮的反应（用户回复或环境执行结果），即作为上一轮奖励计算的 $`s_{t+1}`$。

---

### 3. Binary RL（评估信号 → 标量过程奖励）

#### PRM Judge 构建（多数投票）

给定动作 $`a_t`$ 和 next-state $`s_{t+1}`$，PRM 评估 $`a_t`$ 的质量：

```math
\text{PRM}(a_t,\, s_{t+1}) \;\rightarrow\; r \in \{+1,\,-1,\,0\}
```

运行 $`m`$ 次独立查询取多数票：$`r_{\text{final}} = \text{MajorityVote}(r_1, \ldots, r_m)`$

#### RL 训练目标

以 $`A_t = r_{\text{final}}`$ 为优势，使用非对称裁剪的 PPO 目标：

```math
\rho_t = \frac{\pi_\theta(a_t \mid s_t)}{\pi_{\text{old}}(a_t \mid s_t)},\quad
\mathcal{L}_{\text{pg}} = -\mathbb{E}_t\!\left[\min\!\left(\rho_t A_t,\; \mathrm{clip}(\rho_t,\, 1-\varepsilon,\, 1+\varepsilon_{\text{high}}) \cdot A_t\right)\right]
```

```math
\mathcal{L} = \mathcal{L}_{\text{pg}} + \beta_{\text{KL}} \cdot \mathcal{L}_{\text{KL}}
```

其中 $`\varepsilon = 0.2`$，$`\varepsilon_{\text{high}} = 0.28`$，$`\beta_{\text{KL}} = 0.02`$。

> **注意**：实时对话场景中没有 GRPO 所需的 group 结构，因此不做组内标准化。

---

### 4. Hindsight-Guided On-Policy Distillation（OPD）（指令信号 → Token 级方向监督）

#### 核心思路

若用原始提示加上从 $`s_{t+1}`$ 提取的文字 hint 来增强上下文，**同一个模型**会产生不同的 token 分布——这个"知道了 hint 的模型"就是 teacher。Teacher 与 student 的 per-token log-prob 差值提供方向性优势：
- $`A_t > 0`$：teacher 对该 token 赋予更高概率 → student 应上调
- $`A_t < 0`$：teacher 认为该 token 不合适 → student 应下调

这与 RLHF（标量偏好）、DPO（成对偏好）、标准蒸馏（需要更强 teacher 模型）**本质不同**。

#### 四步流程

**Step 1. Hindsight hint 提取**

```math
\text{Judge}(a_t,\, s_{t+1}) \;\rightarrow\; \bigl\{\text{score} \in \{+1,\,-1\},\; \text{hint} \in \mathcal{T}^*\bigr\}
```

Judge 从 $`s_{t+1}`$ 中蒸馏出简洁、可操作的指令（1~3 句话），而非直接使用原始 next-state（可能噪声大、冗长）。

**Step 2. Hint 筛选与质量过滤**

从正票（score=+1）且 hint 长度 > 10 字符的候选中，选取**最长**（最具信息量）的 hint。若无有效 hint，则**丢弃该样本**（以信号质量换样本数量）。

**Step 3. 增强 Teacher 上下文构建**

将 hint 拼接到最后一条用户消息后：

```math
s_{\text{enhanced}} = s_t \oplus [ \text{user's hint / instruction} ] \backslash n \{ \text{hint} \}
```

**Step 4. Token 级优势计算**

```math
A_t = \log\pi_{\text{teacher}}(a_t \mid s_{\text{enhanced}}) - \log\pi_\theta(a_t \mid s_t)
```

训练仍使用与式（Binary RL）相同的裁剪 PPO 目标，但优势携带了**每个 token 的方向信息**，而非整个序列共享一个标量。

---

### 5. Binary RL + OPD 联合训练

两种方法互补：

| 维度 | Binary RL | OPD | 联合 |
|------|-----------|-----|------|
| 信号类型 | 评估（好/坏） | 方向性 | 评估 + 方向性 |
| 优势粒度 | 序列级标量 | Token 级方向 | 混合 |
| 样本密度 | 所有打分 turn | 仅 hint 通过的 turn | 所有打分 turn |
| 反馈类型 | 隐式反应 / 环境输出 | 显式纠正 | 两者兼顾 |

联合优势计算：

```math
A_t = w_{\text{binary}} \, r_{\text{final}} + w_{\text{opd}} \left( \log \pi_{\text{teacher}}(a_t \mid s_{\text{enhanced}}) - \log \pi_\theta(a_t \mid s_t) \right)
```

默认 $`w_{\text{binary}} = w_{\text{opd}} = 1`$。

---

### 6. 通用 Agentic RL 的 Step-wise Reward

对于长时域任务，仅使用最终结果奖励（outcome reward）导致绝大多数中间步骤无监督信号。

**整合过程奖励与结果奖励**（参考 RLAnything）：

```math
r_{\text{step}} = o + \sum_{i=1}^{m} r_i / m
```

其中 $`o`$ 为最终结果奖励，$`r_i`$ 为 PRM 对第 $`t`$ 步的独立评分。

**优势标准化**：按相同 step index 分组（而非按状态聚类），避免长尾 rollout 导致的不稳定。

## 实验设置

### 支持的智能体场景

| 场景 | 环境 | Next-state Signal | 时域 |
|------|------|-------------------|------|
| OpenClaw（个人） | 用户设备 | 用户回复 / 工具调用结果 | 长 |
| Terminal | Shell 沙箱 | stdout/stderr, exit code | 长 |
| GUI | 屏幕 + 无障碍树 | 视觉状态差异、任务进度 | 长 |
| SWE | 代码仓库 + 测试套件 | 测试结果、diff、lint 输出 | 长 |
| Tool-call | API / 函数执行 | 返回值、错误追踪 | 中 |

### 模型与数据集

| 场景 | 模型 | 训练数据 | 评估数据 |
|------|------|---------|---------|
| Terminal | Qwen3-8B | SETA RL data | 滚动任务平均准确率 |
| GUI | Qwen3VL-8B-Thinking | OSWorld-Verified | 训练集（排除 chrome/multi-apps） |

| 参数 | 值 |
|------|---|

## 实验结果

### 个人智能体：个性化优化效果

**场景 1 — 学生用 OpenClaw 写作业（不想被发现用 AI）**：基础分 0.17

| 方法 | 更新 8 步 | 更新 16 步 |
|------|---------|---------|
| Binary RL | 0.25 | 0.23 |
| OPD | 0.25 | **0.72** |
| **联合（Combined）** | **0.76** | **0.81** |

**关键发现**：
- OPD 因训练样本稀疏，效果显现较慢，但 16 步后大幅领先 Binary RL
- 联合方法始终最优，8 步即达 0.76（基础分 0.17，提升 4.5×）
- 仅需 36 次问题求解交互，智能体即学会避免 AI 化措辞（如 "bold"、过度结构化的步骤式回答），转向更自然的表达风格

**场景 2 — 教师用 OpenClaw 批改作业（希望评语具体且友好）**：
- 仅需 24 次批改交互，评语从简短的"Correct. Well done!"变为详细、鼓励性的多段反馈（含 emoji）

### 通用智能体：四场景 RL 训练均有效提升

Terminal、GUI、SWE、Tool-call 四个场景的 RL 训练曲线均显示持续上升趋势（图 Figure 3）。

### 过程奖励 vs. 结果奖励对比

| 场景 | 整合（Outcome + Process） | 仅结果奖励 |
|------|--------------------------|-----------|
| Tool-call | **0.30** | 0.17 |
| GUI | **0.33** | 0.31 |

过程奖励在 Tool-call 场景提升显著（+76%），在 GUI 场景也有稳定提升，代价是需要额外资源托管 PRM。

## 核心贡献总结

1. **Next-state Signal 作为实时在线学习源**：首次系统性地将 next-state signal 中的评估信号和指令信号统一恢复为在线训练来源，覆盖个人对话、Terminal、GUI、SWE、Tool-call 等异构交互类型。

2. **OpenClaw-RL 基础设施**：首个统一多并发交互流的 Agentic RL 系统。四组件完全解耦异步架构（基于 slime），零中断服务，支持会话感知多轮追踪、优雅权重更新、灵活 PRM 支持和大规模环境并行化。

3. **两种互补的 next-state signal 恢复方法**：
   - **Binary RL**：将评估信号转化为密集标量过程奖励，覆盖所有打分 turn
   - **Hindsight-Guided OPD**：将指令信号转化为 token 级方向优势监督，无需预收集数据、无需外部 teacher、无需成对偏好

4. **实证验证**：个人智能体个性化（36 次交互即显著改善）和通用 Agentic RL（Terminal/GUI/SWE/Tool-call 四场景）均得到验证；过程奖励对长时域任务的重要性得到定量确认。

---

## 与相关工作的关键区别

| 对比维度 | OpenClaw-RL | 现有方法 |
|---------|------------|---------|
| 训练模式 | 实时在线（live online） | 离线批量（offline batch） |
| 信号粒度 | Token 级方向（OPD） | 序列级标量（RLVR/GRPO） |
| 数据来源 | 实时部署交互 | 预收集固定数据集 |
| 多环境统一 | 单框架覆盖 5 种场景 | 各场景专用 pipeline |
| Teacher 来源 | 模型自身（hint 增强上下文） | 需要更强外部 teacher |

---

## 局限性与未来工作

- 个人智能体实验目前为**模拟结果**（用 LLM 模拟用户），真实部署效果待验证
- OPD 的样本效率依赖 hint 质量，在隐式反馈场景（用户仅简短回复）中样本稀疏
- 通用智能体实验仅报告训练曲线，未与 SOTA 基线做系统性对比
- 云端环境托管增加了系统复杂度和资源消耗
