## 研究背景与动机

### 问题背景

量化推理（Quantized Inference）在 LLM 领域已取得显著成效——在保持模型质量的同时大幅提升系统效率。然而，在工业级推荐系统中可靠地应用低精度量化长期以来面临严峻挑战，原因在于：

1. **数值特性差异**：传统推荐模型的权重和激活值具有**高幅度、高方差**的特点，对量化扰动更加敏感
   - 权重均值方差约为 $`10^7`$ 量级，均值 AbsMax 超过 $`10^3`$
   - 激活值方差同样在 $`10^6`$ 量级
2. **硬件利用率低**：传统推荐推理负载通常是内存绑定或控制流绑定，硬件利用率较低，即使硬件支持低精度计算，端到端收益也有限

### 核心洞察

生成式推荐模型（如 OneRec-V2）正在弥合上述差距：
- 依赖更密集的计算路径，提升了硬件利用率
- 训练范式对激活幅度和方差施加了更强的隐式正则化
- 数值分布特性已显著接近 LLM，量化友好度大幅提升

## 数值分布分析（Preliminary）

### 量化基础

给定浮点张量 $`\mathbf{x}`$，量化表示为：

```math
\hat{\mathbf{x}} = Q(\mathbf{x}; s) = \mathrm{round}\left(\frac{\mathbf{x}}{s}\right)
```

其中 $`s`$ 为缩放因子，控制动态范围映射。

### 三类模型对比（权重与激活分布统计，log 尺度）

| 模型 | 权重均值方差 | 权重 AbsMax | 激活均值方差 |
|------|------------|------------|------------|
| 传统推荐模型 | $`\sim 10^7`$ | $`> 10^3`$ | $`\sim 10^6`$ |
| **OneRec-V2** | $`< 0.1`$ | 显著低 | 显著低 |
| Qwen3-8B（LLM） | $`\approx 0.1`$ | $`\approx 2.0`$ | 较低 |

**结论**：OneRec-V2 的数值分布特性远更接近 LLM，而非传统推荐模型，具备高量化友好性。

## 核心方法

本文提出**双组件联合优化**策略：

### 组件一：FP8 PTQ 量化策略

- **量化范围**：仅量化计算密集算子
  - Attention 中的 qkvo 投影层（Linear）
  - Dense FFN 中的线性变换
  - Sparse MoE 中的 grouped GEMM
  - 其余数值敏感或计算量小的组件保持原始精度

- **Linear 层量化**（FP16 vs FP8 对比）：
  - 权重：**通道级量化**（per-channel），离线从 FP16 参数计算 scale
  - 激活：**token 级动态量化**（per-token），运行时计算 scale
  - 计算：FP8 TensorCore 矩阵乘法 + FP32 累加，结果 cast 回 FP16

- **MoE grouped GEMM 量化**：
  - 激活：块级量化，粒度 $`1 \times 128`$（沿最后维度）
  - 权重：块级量化，粒度 $`128 \times 128`$
  - 保留原始路由和并行执行结构

- **存储方式**：所有模型权重预量化后以 `(FP8 weight, FP32 scale)` 对存储于 GPU 内存

### 组件二：RecoGEM 推理基础设施优化

- **直接构建 TensorRT 图**：跳过 PyTorch → ONNX → TensorRT 多阶段转换流程，通过统一算子库直接构建 TensorRT 执行图，实现更紧密的 kernel 选择、算子融合和内存布局控制

- **算子级优化（4项）**：

  | 优化项 | 具体措施 |
  |--------|---------|
  | **量化算子** | 实现高效 per-row 激活量化算子 + 融合低精度矩阵乘法 kernel，减少中间内存流量 |
  | **TopK 优化** | 用基数排序 kernel（RadixTopK）替换默认实现，结合 kernel 融合和 zero-copy 技术 |
  | **Attention 优化** | 针对大 batch、短上下文场景重新设计 attention kernel，引入 batch 级并行和软件流水线 |
  | **MoE 优化** | 使用 Hopper TMA-enabled kernels 和算子融合优化 grouped GEMM，提升 TensorCore 利用率和 MFU |

## 实验结果

### 实验配置

- **模型**：生产级 OneRec-V2，fat-MoE 架构，约 **4B backbone 参数**，每 token 激活约 **0.5B 参数**
- **场景**：单列短视频推荐
- **Batch Size**：32（在线服务配置）
- **基线**：FP16 推理

### 系统性能

| 指标 | FP16 基线 | FP8 优化后 | 提升幅度 |
|------|----------|-----------|---------|
| 端到端延迟 | 139 ms | 70 ms | **-49%** |
| 吞吐量 | 205 | 394 | **+92%** |

### 吞吐量提升分解

| 优化阶段 | 贡献 |
|---------|------|
| 基础设施升级（RecoGEM） | +27% |
| FP8 量化（Linear + MoE） | +42% |
| 算子级优化 | +23% |
| **合计** | **+92%** |

### 在线 A/B 测试（FP8 vs FP16 基线，持续一周）

| 指标 | 快手主站 | 快手 Lite |
|------|---------|---------|
| App 停留时长 | +0.047% | -0.018% |
| 观看时长 | +0.044% | -0.157% |
| 视频播放量 | +0.073% | -0.041% |
| 点赞 | +0.085% | -0.102% |
| 关注 | +0.020% | -0.450% |
| 评论 | +0.805% | +0.261% |
| 收藏 | +0.430% | -0.353% |
| 转发 | +0.313% | +1.047% |

**结论**：在线核心指标无显著下降，FP8 推理实现**零损失部署**。

## 核心贡献总结

1. **数值分析**：系统分析了传统推荐系统低精度推理受限的数值因素和系统因素
2. **关键发现**：实证证明 OneRec-V2 的架构和训练范式转变带来了更可控的权重/激活统计特性，以及更高的硬件利用率，显著提升量化推理可行性
3. **工程实践**：FP8 PTQ 框架 + RecoGEM 优化推理基础设施，实现延迟 -49%、吞吐 +92%，在线指标零损失

---

## 局限性

1. **未探索更低精度**：仅研究 FP8，未涉及 INT8、FP6、FP4 或混合低比特方案，精度-效率边界尚不明确
2. **基础设施要求高**：依赖 TensorRT、Hopper TMA 等硬件能力，对工程资源要求较高，可复现性和可移植性受限
3. **模型覆盖有限**：结论仅基于 OneRec-V2，是否推广至其他生成式推荐架构尚需验证
