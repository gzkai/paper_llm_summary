# 论文精读：MTFM — Meituan Foundation Model for Recommendation

**ArXiv**: https://arxiv.org/pdf/2602.11235

---

## 一、背景与动机

工业级推荐系统通常跨越多个场景（如外卖中的商家推荐、券包推荐、菜品拼单推荐），现有的跨域（CDR）和多场景（MSR）方法存在三大痛点：

1. **Extensibility 不足**：要求不同场景的特征强制对齐，面对数百个异构特征时极易出错且损失信息。
2. **Scalability 不足**：依赖专家手工设计的精巧结构，无法验证 Scaling Law 的收益。
3. **Efficiency 不足**：训练成本与数据量线性增长，多场景数据激增导致训练开销不可承受。

**目标**：设计一个同时具备 Scalability、Extensibility、Efficiency 的推荐基座模型 MTFM（Meituan Foundation Model for Recommendation）。

---

## 二、核心方案（详细）

### 2.1 多场景数据组织：用户粒度聚合

**训练阶段**，将一个时间窗口内所有场景的曝光行为按用户粒度聚合，每个用户对应一个训练样本：

$$X = [\{H_i\}_{i=1}^{N_H}, \{R_i\}_{i=1}^{N_R}, \{U^s\}_{s=1}^{N_S}, \{\{C_i^s, I_i^s\}_{i=1}^{N_E^s}\}_{s=1}^{N_S}]$$

- $H_i$：用户历史行为序列（跨场景共享）
- $R_i$：用户实时行为序列（跨场景共享）
- $U^s$：场景 $s$ 的用户画像特征（场景独立）
- $C_i^s, I_i^s$：场景 $s$ 第 $i$ 次曝光的交叉特征和物品特征（场景独立）

**推理阶段**，各场景部署对应子图，只计算本场景相关的特征，跳过其他场景的计算。

**数据 Pipeline 设计**：
- 特征分为"全场景共享"（H, R）和"场景独立"（U, C, I）两部分
- 先对场景独立特征在各场景内部进行用户粒度聚合，再按列拼接，最后与共享特征按用户粒度合并
- 避免重复计算和冗余存储，显著提升数据吞吐

### 2.2 模型架构：异构 Tokenization + 混合目标注意力

#### 2.2.1 异构 Tokenization

将所有输入特征表示为一个变长异构 Token 序列，共三类 Token：

- **H-token**：历史行为序列中每个 item 对应一个 H-token
  $$\mathbf{h}_{ij} = \text{MLP}_i(\text{Emb}(h_{ij}))$$
  不同历史序列使用不同的 MLP，以适配异构特征空间。所有历史 token 按时间排序，形成 $\mathbf{H} \in \mathbb{R}^{L_H \times d_{\text{model}}}$

- **R-token**：实时行为序列中每个 item 对应一个 R-token，形成 $\mathbf{R} \in \mathbb{R}^{L_R \times d_{\text{model}}}$

- **T-token**：每个曝光行为（候选 item）对应一个 T-token，拼接用户画像、交叉特征和物品侧特征：
  $$\mathbf{t}_i^s = \text{MLP}_s(\text{Emb}(U^s) \| \text{Emb}(C_i^s) \| \text{Emb}(I_i^s))$$
  不同场景使用不同的 MLP，形成 $\mathbf{T} \in \mathbb{R}^{L_T \times d_{\text{model}}}$

三类 Token 拼接为初始输入：
$$\mathbf{X}^{(0)} = (\mathbf{H}; \mathbf{R}; \mathbf{T}) \in \mathbb{R}^{N \times d_{\text{model}}}, \quad N = L_H + L_R + L_T$$

#### 2.2.2 混合目标注意力架构（Hybrid Target Attention, HTA）

模型由 $B$ 个 Block 堆叠，每个 Block 包含 **1 个 Full Attention 层** + **$K$ 个 Target Attention 层**，将复杂度从 $O(N^2)$ 降至 $O\left(\frac{KNL_T + N^2}{K+1}\right)$（$L_T \ll N$）。

**Full Attention 层**：
1. 对输入 Token 做**分组 LayerNorm（GLN）**：不同历史序列的 H-token 分不同组，不同实时序列的 R-token 分不同组，不同场景的 T-token 分不同组
2. 使用 **GQA（Grouped-Query Attention）** 降低显存开销（$H$ 个 Query head，$G$ 个 KV head，$r = H/G$）
3. 应用**动态 Mask** 防止信息泄露，规则如下：
   - H-token 对所有 token 可见
   - R-token 只对时间戳更晚的 token 可见（防止未来信息泄露）
   - T-token 只对自身可见

Full Attention 的计算公式：
$$\mathbf{U}^{(l)}, \{\mathbf{Q}^{(l,h)}\}, \{\mathbf{K}^{(l,g)}, \mathbf{V}^{(l,g)}\} = \text{Split}(\phi_1(f_1^{(l)}(\widetilde{\mathbf{X}}^{(l)})))$$
$$\mathbf{A}^{(l,h)} = \phi_2(\mathbf{Q}^{(l,h)}\mathbf{K}^{(l,g)T} \odot \mathbf{M})\mathbf{V}^{(l,g)}, \quad g = \lceil h/r \rceil$$
$$\mathbf{X}^{(l+1)} = f_2^{(l)}(\text{GLN}(\mathbf{A}^{(l)}) \odot \mathbf{U}^{(l)}) + \mathbf{X}^{(l)}$$

**Target Attention 层**：
- 只更新 T-token 的 embedding，H/R-token 通过 shortcut 直接传递
- Query 来自 T-token，Key/Value 来自全部 Token（复杂度 $O(L_T \cdot N)$）

Target Attention 的计算公式：
$$\mathbf{U}_T^{(l)}, \{\mathbf{Q}_T^{(l,h)}\} = \text{Split}(\phi_1(f_{uq}^{(l)}(\widetilde{\mathbf{X}}_T^{(l)})))$$
$$\{\mathbf{K}^{(l,g)}, \mathbf{V}^{(l,g)}\} = \text{Split}(\phi_1(f_{kv}^{(l)}(\widetilde{\mathbf{X}}^{(l)})))$$
$$\mathbf{A}_T^{(l,h)} = \phi_2(\mathbf{Q}_T^{(l,h)}\mathbf{K}^{(l,g)T} \odot \mathbf{M}_T)\mathbf{V}^{(l,g)}$$
$$\mathbf{X}^{(l+1)} = (\mathbf{X}^{(l)}[:L_H+L_R]; \mathbf{X}_T^{(l+1)})$$

最终，最后一层的 T-token embedding 输入 **MMoE** 模块，计算多场景多目标的预估分。

### 2.3 训练与推理优化

#### 训练优化

**CPU-GPU Pipeline 优化**：
- 推荐系统中大量特征存于 CPU，导致 CPU-GPU 频繁同步阻塞
- 通过消除同步点实现 CPU-GPU 执行重叠
- 将多次 D2D 拷贝合并为单次原子操作
- 效果：训练吞吐提升 **20%**

**自定义 Kernel 开发**：
- **增强版 FlashAttention-2**：针对动态 Mask 的非连续内存访问问题，构造连续对齐的 Mask 内存布局，前向缓存 Mask tile，反向实现 Mask 与中间变量分时复用，将 Mask 加载延迟隐藏在计算流水线中
- **Triton 融合算子**：将 Group Layer Normalization（GLN）和动态 Mask 构建融合为单一算子，消除中间结果反复读写，利用向量化内存访问提升带宽利用率
- 效果：额外提升吞吐 **57%**

#### 推理优化

**结构化稀疏（Structured Sparsity）**：
- 利用 NVIDIA Ampere 架构的稀疏张量核心，对 HSTU 的线性投影层（Q/K/V 及输出层）实施 2:4 结构化剪枝
- 效果：显存减少 50%，推理吞吐提升约 **10%**，延迟降低约 **0.2ms**

**注意力计算剪枝**：
- 动态剔除 Padding 产生的无效计算，结合业务先验知识屏蔽无效注意力权重
- 效果：推理吞吐提升约 **5%**

**场景化子图部署**：
- MTFM 统一训练但分场景部署，将全局计算图拆解为场景专属子图，消除无关场景特征的冗余计算
- 结合 BF16 半精度推理和 M-Falcon 批处理算法进一步提升推理吞吐

## 三、实验结果

### 数据集
美团外卖真实工业数据，三个场景：
- **HP**（首页商家推荐）：流量最大，指标 CTR & CTCVR
- **SQS**（神抢手券包推荐）：指标 CTR & CTCVR & IMD & WRITE
- **PHF**（拼好饭菜品推荐）：指标 CTR & CTCVR

### 离线实验（RQ1）
与 DCNv2、MMoE、RankMixer、MTGR、OneTrans、STAR、PEPNet 对比：
- MTFM 在几乎所有场景和指标上达到 SOTA
- 有效缓解多场景的"跷跷板效应"（如 PEPNet 在 HP/PHF 好但 SQS 差）
- CTR 任务平均 GAUC 提升 **0.36pp**，最高 **0.76pp**
- CTCVR 任务平均 GAUC 提升 **0.29pp**，最高 **0.53pp**

### HTA 效率分析（RQ2）
| 配置 | GAUC 变化 | 训练吞吐 |
|------|----------|---------|
| Full Attention（baseline） | 0 | 1x |
| (1:1)×B | ≈0（无损） | 显著提升 |
| **(3:1)×B + GQA（最终选择）** | ≈0（无损） | **2x** |
| (5:1)×B | -0.07pp | 更快 |
| Target Attention Only | -0.12pp | 最快 |

### Scaling Law 验证（RQ3）
- 模型规模从 10x 到 70x（以 MMoE 为基线），三个场景均呈现一致的 Scaling Law 曲线（$R^2 > 0.93$）
- 随训练 token 数增加，不同规模模型的性能差距逐渐扩大，大模型优势持续增强

### 可解释性分析（RQ4）
注意力热力图显示：
- 各场景 T-token 都能有效聚合来自全场景 H-token 的信息
- HP 和 SQS 的 T-token 更关注自身场景的 H-token，PHF 的 T-token 更依赖 PHF 历史
- 呈现清晰的块状注意力模式，验证模型具备场景感知能力

### 在线 A/B 实验（RQ5）
| 场景 | CTR | UV_CTCVR | 订单量 | 延迟 |
|------|-----|----------|--------|------|
| SQS  | +1.89% | +2.46% | **+2.98%** | -5ms |
| PHF  | +1.53% | +1.03% | **+1.45%** | -6ms |

在线订单提升相当于该业务 2-3 轮模型迭代的累计收益。

---

## 四、总结

MTFM 是美团面向工业级多场景推荐的基座模型，核心创新点：

1. **异构 Tokenization**：无需特征对齐，用不同 MLP 将各场景特征映射到统一 token 空间，保证 Extensibility
2. **混合目标注意力（HTA）**：Full Attention + Target Attention 交错，配合 GQA，在保持建模能力的同时实现 2x 训练加速，保证 Scalability
3. **系统级协同优化**：CPU-GPU Pipeline 优化 + 自定义 Kernel + 结构化稀疏 + 场景化子图部署，训练推理吞吐大幅提升，保证 Efficiency
