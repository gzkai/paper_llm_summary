## 研究背景与动机

### 问题定义

**生成式推荐（Generative Recommendation, GR）** 是近年兴起的推荐新范式：将物品表示为离散语义 ID（Semantic IDs，通过 RQ-VAE 或 Residual K-means 量化得到），基于用户历史序列 $`X_u = [x_1, \ldots, x_{t-1}]`$ 自回归地预测下一个物品的 Semantic ID 序列。

GR 具有强大的序列推理能力，但在**超长用户行为序列**建模上存在严重瓶颈：

### 三大核心问题

1. **序列长度壁垒（Sequence Length Barrier）**：GR 的高计算开销迫使工业部署的序列长度通常不超过 10,000。而用户最近 10,000 条交互通常仅覆盖数月历史，无法捕捉终身行为模式。

2. **低效的序列扩展（Inefficient Sequence Scaling）**：实验表明 GR 从增加序列长度获得的性能增益是次线性的（sub-linear）。在严格延迟约束下，Transformer 架构的 GR 在长序列场景下的 ROI 极低。

3. **注意力的近因偏差（Recency Bias in Attention）**：GR 的注意力机制倾向于过度关注用户序列的最近片段，削弱了对长期历史的有效利用。

---

## 核心方法：GEMs

GEMs 是一个基于 **Encoder-Decoder 范式**的多流框架，通过多时间尺度视角捕捉用户终身交互序列。

### 1. 序列分段（Sequence Segmentation）

设用户 $`u`$ 的终身序列为 $`X_u = [x_1, \dots, x_T]`$（$`x_1`$ 最旧，$`x_T`$ 最新，$`T`$ 可超过 100,000），引入两个阈值 $`1 < T_{\text{lifecycle}} < T_{\text{recent}} < T`$，将序列分为三段：

```math
\begin{aligned}
    X_{u,\ \text{recent}} &= X_u[T_{\text{recent}}:\ ]\\
    X_{u,\ \text{mid-term}} &= X_u[(T_{\text{lifecycle}}+1):T_{\text{recent}}]\\
    X_{u,\ \text{lifecycle}} &= X_u[:\ T_{\text{lifecycle}}]
\end{aligned}
```

工业部署配置：$`T_{\text{recent}} = 256`$，$`T_{\text{lifecycle}} = 5000`$。

---

### 2. 三流编码器（Multi-Stream Encoders）

#### 2.1 Recent Stream（近期流）

提取多模态特征（视频 ID、作者 ID、标签、时间戳、播放时长、视频时长、交互标签），拼接后经 MLP 投影：

```math
\begin{aligned}
    f_{\text{recent}}^{i} &= [e_{vid}^{i}, e_{aid}^{i}, e_{tag}^{i}, e_{ts}^{i}, e_{pt}^{i}, e_{dur}^{i}, e_{label}^{i}]\\
    h_{recent}^{i} &= \text{MLP}(f_{\text{recent}}^{i}),\ i \in [1, T_{\text{recent}}]
\end{aligned}
```

再通过多层自注意力（带位置编码）捕捉用户即时兴趣。

#### 2.2 Mid-term Stream（中期流）

中期序列通常包含数千 token，采用**轻量级索引器（Lightweight Indexer）**进行稀疏交叉注意力，详见第 3 节。

#### 2.3 Lifecycle Stream（终身流）

受 VISTA 启发，采用**两阶段机制**：

- **离线阶段**：用带 QLU 优化的准线性注意力（Quasi-Linear Attention, QLA）压缩终身历史：

```math
\text{QLUAttn}(T, E) = \varphi(Q[E])\varphi\!\left(\varphi(K[T])^\top V[T]\right) + \Delta(\varphi(Q[E]), \varphi(K[T]))V[T]
```

压缩过程通过生成重建损失监督：

```math
\mathcal{L}_{\text{recon}} = \sum_{i=1}^{M-1}\left\|v_i - u_{i+1}\right\|_2^2
```

其中 $`v_i`$ 和 $`u_{i+1}`$ 分别为压缩和重建的表示。

- **在线阶段**：压缩表示从低延迟 KV 系统检索，经轻量自注意力模块处理后供解码器使用。

---

### 3. 轻量级索引器（Lightweight Indexer for Cross Attention）

**动机**：中期序列的全量交叉注意力计算代价高且噪声大，需要在保持信号质量的同时降低复杂度。

**核心思路**：训练一个轻量级索引器（使用远少于主模型的注意力头 $`H_{\text{indexer}} \ll H`$）来模拟全量交叉注意力的分布，推理时仅对 Top-K 个重要 token 做稀疏注意力。

#### 3.1 全量注意力分数（Teacher）

在第 $`j`$ 个 Block 的第 $`h`$ 个头，计算全量交叉注意力分数：

```math
\begin{aligned}
    Q^{(j,h)}_{\text{mid-term}} &= W^{(j,h)}_{Q, \text{mid-term}} H^{(j-1)}_{\text{mid-term}}\\
    K^{(j,h)}_{\text{mid-term}} &= W^{(j,h)}_{K, \text{mid-term}} H^{(j-1)}_{\text{mid-term}}\\
    S^{(j,h)}_{\text{mid-term}} &= {Q^{(j,h)}_{\text{mid-term}}}^\top K^{(j,h)}_{\text{mid-term}} / \sqrt{d_k}\\
    S^{(j)}_{\text{mid-term}} &= \sum_{h=1}^H S^{(j,h)}_{\text{mid-term}}
\end{aligned}
```

#### 3.2 索引器分数（Student）

用少量头 $`H_{\text{indexer}}`$ 计算近似分数，并通过额外投影 $`D^{(j)}`$ 聚合：

```math
\begin{aligned}
    Q^{(j,h)}_{\text{indexer}} &= W^{(j,h)}_{Q, \text{indexer}} H^{(j-1)}_{\text{mid-term}}\\
    K^{(j,h)}_{\text{indexer}} &= W^{(j,h)}_{K, \text{indexer}} H^{(j-1)}_{\text{mid-term}}\\
    S^{(j,h)}_{\text{indexer}} &= {Q^{(j,h)}_{\text{indexer}}}^\top K^{(j,h)}_{\text{indexer}} / \sqrt{d_k}\\
    D^{(j)} &= W^{(j)}_{D} H^{(j-1)}_{\text{mid-term}}\\
    S^{(j)}_{\text{indexer}} &= {D^{(j)}}^\top [S^{(j,1)}_{\text{indexer}}; \ldots; S^{(j,H_{\text{indexer}})}_{\text{indexer}}]
\end{aligned}
```

#### 3.3 KL 蒸馏损失

通过 KL 散度对齐索引器分数与全量注意力分数：

```math
\mathcal{L}_{indexer} = \sum_{j=1}^{L} D_{KL}\!\left(S^{(j)}_{\text{indexer}}\ |\ \text{Softmax}(S^{(j)}_{\text{mid-term}})\right)
```

#### 3.4 稀疏注意力（推理阶段）

基于训练好的 $`S^{(j)}_{\text{indexer}}`$，每个 Query 只对 Top-K 个 Key/Value 做稀疏注意力，计算复杂度从 $`\mathcal{O}(n_q n_m d)`$ 降至 $`\mathcal{O}(n_q K d)`$（$`K \ll n_m`$）：

```math
\begin{aligned}
    H^{(j)\prime}_{\text{mid-term}} &= H^{(j-1)}_{\text{mid-term}} + \text{SparseCrossAttn}(H^{(j-1)}_{\text{mid-term}},\ i \in \text{TopK}(S^{(j)}_{\text{indexer}}))\\
    H^{(j)}_{\text{mid-term}} &= H^{(j)\prime}_{\text{mid-term}} + \text{FFN}(\text{RMSN}(H^{(j)\prime}_{\text{mid-term}}))
\end{aligned}
```

---

### 4. 多流解码器与参数无关融合（Multi-Stream Decoder & Parameter-Free Fusion）

解码器自回归生成目标物品的 Semantic IDs，初始输入为：

```math
D^{(0)} = [E_{\text{BOS}};\ E_{s^{1}_{x_t}};\ \ldots;\ E_{s^{d}_{x_t}}]
```

每个 Block $`j`$ 先做因果自注意力：

```math
D^{(j)\prime} = D^{(j-1)} + \text{CausalSelfAttn}(D^{(j-1)})
```

然后对三个流分别做独立参数的交叉注意力：

```math
\begin{aligned}
    D^{(j)}_{s} &= D^{(j)\prime} + \text{CrossAttn}(D^{(j)\prime},\ H^{(L_s)}_{s},\ H^{(L_s)}_{s})\\
    D^{(j)*}_{s} &= D^{(j)}_{s} + \text{FFN}(\text{RMSN}(D^{(j)}_{s}))
\end{aligned}
```

其中 $`s \in \{\text{recent}, \text{mid-term}, \text{lifecycle}\}`$。

**参数无关融合**（直接求和，无可学习权重）：

```math
D^{(j)} = D^{(j)*}_{\text{recent}} + D^{(j)*}_{\text{mid-term}} + D^{(j)*}_{\text{lifecycle}}
```

> **为什么不用可学习融合？** 终身序列与近期序列存在分布偏移，可学习权重（门控/MLP）会使模型过度关注近期流，抑制中期和终身流的贡献。实验可视化证实了这一点（注意力权重几乎全部分配给近期流）。

---

### 5. 三阶段训练策略

| 阶段 | 描述 | $`\lambda`$ |
|------|------|------------|
| **Stage 1** | 全量注意力训练收敛，索引器 token 选择禁用 | 0 |
| **Stage 2** | $`\mathcal{L}_{\text{indexer}}`$ 启用，对齐索引器与全量注意力分数 | 1 |
| **Stage 3** | 激活 Top-K 稀疏 token 选择，使训练与推理对齐 | 1 |

总训练损失：

```math
\mathcal{L} = \mathcal{L}_{\text{NTP}} + \lambda \mathcal{L}_{indexer}
```

其中 $`\mathcal{L}_{\text{NTP}} = -\log p_\theta(s^d_{x_t} \mid \ldots)`$ 为下一 token 预测损失。

> **设计要点**：索引器在训练时与主模型梯度解耦，仅由 $`\mathcal{L}_{indexer}`$ 优化，防止索引器的推理能力泄漏到交叉注意力模块，避免训练-推理不一致。

### 主实验（整体性能对比）

| 方法 | Seq Len | Recall@100 | NDCG@100 | Recall@500 | NDCG@500 | Recall@1000 | NDCG@1000 |
|------|---------|-----------|---------|-----------|---------|------------|---------|
| TIGER | 1024 | 0.1282 | 0.0053 | 0.2351 | 0.0104 | 0.2847 | 0.0139 |
| GRank | 1024 | 0.1010 | 0.0047 | **0.2396** | **0.0109** | **0.3178** | **0.0142** |
| **GEMs（本文）** | **Lifelong** | **0.1553** | **0.0064** | **0.2911** | **0.0122** | **0.3511** | **0.0145** |
| **提升** | — | **+21.14%** | **+20.75%** | **+21.49%** | **+11.93%** | **+10.47%** | **+2.11%** |

GEMs 在所有指标上全面超越最优基线（GRank / TIGER）。

### 消融实验（各流贡献）

| 配置 | Hrecall@Level1@1000 | Hrecall@Level2@1000 | Hrecall@Level3@1000 |
|------|-------------------|-------------------|-------------------|
| (1) 仅 Recent | 0.9311 | 0.4093 | 0.2660 |
| (2) Recent + Mid-term | 0.9372 | 0.4322 | 0.3310 |
| (3) Recent + Lifecycle | 0.9420 | 0.4290 | 0.3210 |
| **(4) 完整 GEMs** | **0.9420** | **0.5115** | **0.3817** |

- Mid-term 流对第一层级召回贡献最大
- Lifecycle 流对深层级（Level 2/3）召回贡献最大
- 三流缺一不可

### 融合策略对比

| 策略 | 描述 | Hrecall@L1 | Hrecall@L2 | Hrecall@L3 |
|------|------|-----------|-----------|-----------|
| (a) 全拼接，共享编解码器 | — | 0.9317 | 0.4750 | 0.3412 |
| (b) 独立编码器，输出拼接 | — | 0.9368 | 0.4834 | 0.3603 |
| (c) 独立编解码器 + 门控权重 | — | 0.9360 | 0.4819 | 0.3537 |
| **(d) 参数无关求和（本文）** | — | **0.9420** | **0.5115** | **0.3817** |

可视化分析证实：策略 (a)(b)(c) 中注意力权重大量集中于近期流，参数无关融合是唯一能让三流均等贡献的方案。

### 索引器效率对比

| 聚合方法 | $`T_{\text{recent}}`$ | $`T_{\text{lifecycle}}`$ | 推理延迟 (ms) | Recall@1000 |
|---------|---------------------|------------------------|------------|------------|
| Brute-force | 1000 | 2000 | 60.18 | 0.3312 |
| Brute-force | 1000 | 5000 | 103.68 | 0.3464 |
| **Indexer（本文）** | 1000 | 5000 | **32.2** | **0.3511** |

索引器在更长序列（5000 vs 2000）上实现更低延迟（32.2ms vs 60.18ms），同时性能更优——稀疏注意力的降噪效果使其超越全量暴力注意力。

## 核心贡献总结

1. **GEMs 多流框架**：首个在 GR 范式下对用户终身序列进行统一建模的框架，通过三流（Recent / Mid-term / Lifecycle）分治策略突破序列长度壁垒

2. **轻量级索引器（Lightweight Indexer for Cross Attention）**：通过 KL 蒸馏训练少头索引器模拟全量注意力分布，推理时仅对 Top-K token 做稀疏注意力，在 $`T_{\text{lifecycle}} = 5000`$ 时延迟仅 32.2ms，同时降噪效果使性能超越全量注意力

3. **参数无关融合（Parameter-Free Fusion）**：通过直接求和避免可学习权重导致的近因偏差，使三流均等贡献，在所有融合策略对比中最优

4. **工业验证**：快手 & 快手极速版 A/B 测试全面提升用户使用时长和观看时长，并通过 TensorRT + User Batching 等工程优化满足高并发延迟约束

---

## 局限性与未来工作

- 当前 Lifecycle 流依赖离线压缩，对用户兴趣的实时漂移响应可能存在延迟
- 序列分段阈值 $`T_{\text{recent}}`$ 和 $`T_{\text{lifecycle}}`$ 为全局统一设置，未针对个体用户活跃度自适应调整
- 多模态特征（图像、音频）在 Lifecycle 流的压缩中尚未充分利用
