## 研究背景与动机

现有生成式推荐系统（如 OneRec 系列）已将多阶段推荐管线统一为端到端生成框架，但与通用智能之间仍存在显著差距：

1. **数据孤岛**：推荐模型被隔离数据所束缚，无法像 LLM 那样利用大规模数据涌现能力
2. **领域专家局限**：系统只会协同过滤模式匹配，缺乏世界知识、推理能力和指令跟随
3. **评测体系缺失**：现有 benchmark 局限于单一任务，无法全面评估推荐基础模型的综合能力

**本文目标**：提出统一框架，将 LLM 的预训练/后训练范式迁移到推荐系统，并建立第一个全面的推荐指令跟随 benchmark。

## 核心贡献

### 贡献一：RecIF-Bench（推荐指令跟随基准）

覆盖 **8 个任务、4 层能力体系**，是首个全面评估推荐基础模型的 benchmark：

| 层级 | 任务 | 输入 | 输出 | 评测指标 |
|------|------|------|------|---------|
| **L0：语义对齐** | 物品理解 | Item $i$ | 文字描述 | LLM-as-Judge |
| **L1：基础推荐** | 短视频推荐 | 历史 $\mathcal{H}^{video}$ | 下一个视频 | Pass@1/32, Recall@32 |
| **L1：基础推荐** | 广告推荐 | $\mathcal{H}^{video}+\mathcal{H}^{ad}$ | 下一个广告 | Pass@1/32, Recall@32 |
| **L1：基础推荐** | 商品推荐 | $\mathcal{H}^{video}+\mathcal{H}^{product}$ | 下一个商品 | Pass@1/32, Recall@32 |
| **L1：基础推荐** | 行为标签预测 | 历史 + 候选物品 | Yes/No | AUC |
| **L2：指令跟随** | 交互式推荐 | 用户画像 + 自然语言 Query | 物品 | Pass@1/32, Recall@32 |
| **L2：指令跟随** | 条件推荐 | 历史 + 目标行为标签 | 对应物品 | Pass@1/32, Recall@32 |
| **L3：推理** | 推荐解释 | 画像 + 历史 + 物品 | 自然语言解释 | LLM-as-Judge |

**数据集规模**：
- 约 1.2 亿次交互，来自 20 万用户，覆盖短视频、广告、商品三个领域
- 公开训练集：9600 万次交互，16 万用户

## 方法框架

### 基础设定：物品即 Token

将物品视为独立模态，采用 **Itemic Tokens**：

1. 用 Qwen3-8B-Embedding 提取物品语义嵌入
2. 通过 **RQ-Kmeans** 三层量化（每层 codebook 大小 8192），将物品映射为离散码：
   ```
   <|item_begin|><item_a_5028><item_b_6733><item_c_2559><|item_end|>
   ```
3. 统一词表 $`\mathcal{V} = \mathcal{V}_{text} \cup \mathcal{V}_{item}`$

**推荐即自回归生成**：给定指令 $`\mathcal{I}`$ 和用户上下文 $`\mathcal{C}`$，最大化目标序列 $`Y`$ 的似然：

$$
\mathcal{L}(\theta) = - \sum_{t=1}^{|Y|} \log P_{\theta}(y_{t} \mid \mathcal{I}, \mathcal{C}, y_{\lt t})
$$

### 预训练（Pre-Training）

#### 数据构成

三类推荐域数据：

| 数据类型 | 作用 |
|---------|------|
| **Itemic Dense Caption** | 给定物品 token → 生成文字描述，建立模态语义桥 |
| **Sequential User Behavior** | 用户行为序列 → 下一物品预测，注入协同过滤信号 |
| **Interleaved User Persona Grounding** | 自然语言 + 物品 token 交织的用户画像，深度语义接地 |

混合通用域数据（数学、代码、医学等），防止灾难性遗忘。

#### 两阶段训练

**Stage 1：Itemic-Text Alignment（冻结预训练参数）**
- 仅训练 itemic token 的 embedding 参数（+ 大模型的 output projection）
- 在 Qwen3 冻结参数基础上建立稳定的语义对齐

**Stage 2：Full-Parameter Co-Pretraining（全参数训练）**
- 解冻全部参数，注入推荐知识
- 保持通用域数据混合比例，防止遗忘

**训练超参**：AdamW，$`\beta_1=0.9, \beta_2=0.95`$，weight decay $`0.1`$；cosine decay 学习率，Stage 1 峰值 $`10^{-3}`$，Stage 2 峰值 $`10^{-4}`$；最大上下文长度 **32K tokens**。

#### 推荐域 Scaling Laws

遵循 Chinchilla 方法论，拟合参数化损失函数：

```math
L(N, D) = E + \frac{A}{N^{\alpha}} + \frac{B}{D^{\beta}}
```

拟合结果：

```math
L(N, D) = 0.4232 + \frac{502.32}{N^{0.3325}} + \frac{7.02}{D^{0.1865}}
```

最优 Scaling 指数：

```math
N_{\text{opt}} \propto C^{0.44}, \quad D_{\text{opt}} \propto C^{0.56}
```

**关键发现（与 Chinchilla 对比）**：
- 推荐域是**数据密集型 Scaling**（$`b=0.56 > a=0.44`$），需要比通用文本更激进地扩展数据量
- 数据指数 $`\beta=0.19`$ 远低于通用文本（$`\approx 0.28`$），归因于 Qwen3 backbone 的迁移学习效果
- 不可约损失 $`E=0.42`$ 远低于自然文本（$`\approx 1.69`$），推荐任务具有更低的固有熵

### 后训练（Post-Training）

三阶段流程：

#### Stage 1：多任务 SFT

- 恢复指令跟随和推理能力
- 混合推荐任务指令对 + 通用域高质量推理数据
- 学习率 $`2\times10^{-5}` 到 $`5\times10^{-6}`$

#### Stage 2：通用能力 On-policy Distillation

针对预训练引入的通用能力退化，以原版 Qwen3 为教师模型进行**在线蒸馏**：

```math
\mathbb{D}_{KL}(\pi_\theta \| \pi_{\text{teacher}}) = \mathbb{E}_{x \sim \pi_\theta}\left[\log\pi_\theta(x_{t+1}|x_{1..t}) - \log\pi_{\text{teacher}}(x_{t+1}|x_{1..t})\right]
```

策略梯度目标：

$$
\nabla_\theta J(\theta) = \mathbb{E}_{o \sim \mathcal{D}, x \sim \pi_\theta} \left[ \sum_{t=1}^{T} \nabla_\theta \log \pi_\theta (x_t \mid o, x_{\lt t}) \cdot R_{KL}(o, x) \right]
$$

裁剪奖励防止训练不稳定：

```math
R_{KL}(o, x) = \text{clip}\left(-\mathbb{D}_{KL}(\pi_\theta \| \pi_{\text{teacher}}), \alpha, \beta\right)
```

**处理词表不匹配**：对出现 itemic token 的轨迹，将教师 log 概率设为 $`-10^9`$ 并截断，配合高温度采样探索。

#### Stage 3：推荐强化学习（Rec-RL）

采用 **GRPO**（Group Relative Policy Optimization）：

```math
\mathcal{L}_{GRPO}(\theta) = \frac{1}{G}\sum_{i=1}^{G}\left(\text{Adv}_i \cdot \log\pi_\theta(R_i|q)\right) - \beta \cdot KL(\pi_\theta \| \pi_{ref})
```

基于规则的稀疏命中奖励：

```math
r(R_i) = \begin{cases} +1.0 & \text{if target itemic token } s \in R_i \\ 0.0 & \text{otherwise} \end{cases}
```

## 实验结果

### RecIF-Bench 主要结果（部分核心指标）

| 任务 | 指标 | LC-Rec-8B | OneRec-1.7B | OneRec-8B | OneRec-8B-Pro |
|------|------|-----------|-------------|-----------|---------------|
| 短视频推荐 | Pass@1 | 0.0341 | 0.0496 | 0.0542 | **0.0548** |
| 短视频推荐 | Recall@32 | 0.0180 | 0.0272 | 0.0355 | **0.0369** |
| 广告推荐 | Pass@1 | 0.0197 | 0.0169 | 0.0219 | **0.0259** |
| 广告推荐 | Recall@32 | 0.0723 | 0.0707 | 0.0877 | **0.0964** |
| 交互式推荐 | Pass@32 | 0.3730 | 0.3170 | 0.4570 | **0.5080** |
| 行为标签预测 | AUC | 0.6139 | 0.6184 | 0.6615 | **0.6912** |
| 推荐解释 | LLM-Judge | **3.9350** | 3.3540 | 3.6774 | 4.0381 |

**规律**：数据 Scaling（Pro > Standard）和模型 Scaling（8B > 1.7B）均带来一致提升。

### 通用能力保留（Thinking 模式，8B）

| 指标 | Qwen3-8B | OneRec-8B | OneRec-8B-Pro |
|------|---------|-----------|---------------|
| MATH-500 | **0.952** | 0.946 | 0.938 |
| GSM8K | 0.957 | **0.958** | **0.958** |
| AIME'24 | **0.792** | 0.725 | 0.719 |
| MMLU-Pro | **0.724** | 0.534 | 0.520 |
| GPQA-Diamond | **0.561** | 0.500 | 0.505 |
| IFEVAL | **0.858** | 0.789 | 0.763 |

数学推理保留较好，通用知识（MMLU-Pro）有一定下降，属于推荐专业化与通用能力的权衡。

### Amazon 跨域迁移（10 个数据集平均）

OneRec-Foundation 在 10 个 Amazon 数据集上均取得 SOTA，Recall@10 平均超过次优基线 **26.8%**。

**三种迁移适配策略对比**：

| 策略 | 碰撞率 | 优势 |
|------|--------|------|
| Extended Residual Quantization | 3.05% | 保留协同过滤信号 |
| Text-Only Adaptation | 4.27% | 语言理解能力强 |
| **Text-Augmented Itemic Tokens** | **0.47%** | **最优，兼顾两者** |

**少样本优势**：10% 数据下，OneRec-Foundation 比 TIGER 高 **219.7%**（全量数据下为 77.7%）；OneRec 保留全量性能的 45.2%，TIGER 仅保留 23.0%。

### 后训练各阶段消融（8B 模型）

| 阶段 | 推荐任务 | 通用能力 |
|------|---------|---------|
| Stage 1 SFT | 推荐能力恢复，但通用能力下降 | MMLU-Pro 0.53 |
| Stage 2 On-policy Distillation | 推荐能力略降 | MMLU-Pro 回升至 0.55 |
| Stage 3 Rec-RL | 推荐能力显著提升 | 通用能力轻微下降 |

## 核心贡献总结

1. **RecIF-Bench**：首个覆盖 8 任务 4 层能力体系的推荐指令跟随 benchmark，同时评估通用智能保留
2. **开源全栈训练框架**：数据处理 + 协同预训练 + 后训练全流程开源，验证推荐域 Scaling Laws
3. **OneRec-Foundation 模型**（1.7B & 8B）：在 RecIF-Bench 全任务 SOTA，Amazon 迁移平均提升 26.8%
4. **三阶段后训练策略**：SFT + On-policy Distillation + Rec-RL，实现推荐精度与通用智能的平衡

---

## 局限性与未来工作

1. **Tokenizer 迁移瓶颈**：预训练 tokenizer 在窄域（如 Amazon）碰撞率高，需要更好的物品索引方案
2. **数据混合策略**：通用域数据混合比例和利用效率有待优化，以更好平衡通用能力与推荐专业化
3. **Chain-of-Thought 推理**：CoT 推理目前仅在有限场景有效，需要探索更系统的测试时 Scaling 策略
