## 研究背景与动机

### 现有生成式推荐的局限

以 OneRec 为代表的生成式推荐框架（GR）通过自回归解码物品 ID（Itemic Token）实现端到端推荐，已在快手等工业系统中取得显著进展。然而，这类模型本质上是**隐式预测器**：

- 缺乏**显式、可验证的推理路径**（Chain-of-Thought）
- 无法像现代 LLM 那样进行可解释的多步推理
- 面对嘈杂的工业级用户行为序列时，推理能力有限
- 无法动态响应用户对话指令（如"帮我推荐轻松的内容"）

### 核心目标

将 LLM 的显式推理能力（CoT）与个性化生成推荐无缝融合于单一统一模型，实现：
1. 可解释的推理路径生成
2. 更高的推荐准确率
3. 对话式动态调整推荐

## 问题定义

### Itemic Token

每个物品 $`v`$ 被映射为一组语义丰富的离散 token 序列 $`\mathbf{s}_v = (s_v^1, \dots, s_v^L)`$，由物品的多模态和协同内容生成。

### 标准生成式推荐（隐式）

```math
\mathbf{s}_{v_{n+1}} \sim P(\cdot|\mathbf{s}_{v_1}, \dots, \mathbf{s}_{v_{n}};\theta)
```

### OneRec-Think 的推理增强推荐（显式）

```math
\mathbf{\tau} \sim P\left(\cdot \mid \mathcal{P}(\mathbf{s}_{v_1}, \dots, \mathbf{s}_{v_n}); \theta\right)
```

```math
\mathbf{s}_{v_{n+1}} \sim P\left(\cdot \mid \mathcal{P}(\mathbf{s}_{v_1}, \dots, \mathbf{s}_{v_n}), \mathbf{\tau}; \theta\right)
```

先生成推理路径 $`\mathbf{\tau} = (r_1, \dots, r_M)`$，再基于推理路径自回归生成目标物品 token，两阶段在单次前向传播中完成。

## 核心方法：三阶段框架

### 阶段一：Itemic Alignment（物品语义对齐）

**目标**：将推荐系统的物品语义空间与 LLM 的文本嵌入空间对齐，使 LLM 能够"理解"离散的 Itemic Token。

#### 四类预训练任务

| 任务 | 描述 | 数据占比 |
|------|------|---------|
| **Interleaved User Persona Grounding** | 将 Itemic Token 与用户 Persona 文本交织，静态属性+搜索行为+交互序列+兴趣摘要 | 24.30% |
| **Sequential Preference Modeling** | 给定用户最近 256 条交互序列，预测下一个物品 | 65.73% |
| **Itemic Dense Captioning** | 从 Itemic Token 生成物品的文字描述，建立 token↔语义的双向映射 | 4.94% |
| **General Language Modeling** | 通用语料预训练，保持 LLM 基础语言能力 | 5.03% |

#### 两阶段训练策略

**Token Warm-up 阶段**：
- 冻结 LLM backbone，仅训练新增 Itemic Token 的 embedding
- 学习率 $`5 \times 10^{-4}`$（较高，使随机初始化的 embedding 快速收敛）
- 仅在 Interleaved User Persona Grounding 任务上训练
- 工业设置：使用 60 亿 token 的语料进行充分语义对齐

**Multi-Task Integration 阶段**：
- 解冻全部参数（工业场景使用 LoRA 高效微调）
- 四类任务联合优化，防止模型将 Itemic Token 退化为无语义的 ID
- 学习率 $`3 \times 10^{-4}`$

---

### 阶段二：Reasoning Activation（推理激活）

**挑战**：工业级用户行为序列噪声大、长度长，直接应用 CoT 效果差。

**解决方案**：两步蒸馏策略，从精简上下文中提取推理轨迹，再迁移到嘈杂真实序列。

#### Step 1：Bootstrapping with Pruned Contexts（基于精简上下文的推理引导）

对每个用户，用相似度函数 $`g(\cdot,\cdot)`$ 检索与目标物品最相关的 top-$k$（$k=10$）历史物品：

```math
g((\mathbf{s}_{v_1},\dots,\mathbf{s}_{v_{n}}), \mathbf{s}_{v_{n+1}}) = (\mathbf{s}_{w_1},\dots,\mathbf{s}_{w_k})
```

用精简后的序列提示已对齐的模型生成高质量推理路径 $`\mathbf{\tau}`$：

```math
\mathbf{\tau} \sim P\left(\cdot \mid \mathcal{P}_{r}((\mathbf{s}_{w_1},\dots,\mathbf{s}_{w_k}), \mathbf{s}_{v_{n+1}}); \theta\right)
```

#### Step 2：Learning to Reason from Noisy Sequences（从嘈杂序列中学习推理）

用精简推理路径作为监督信号，训练模型从完整嘈杂序列中进行推理，联合优化目标：

```math
\mathcal{L}_{\text{RA}} = - \Biggl(
     \sum_{i=1}^{M} \log P(r_{i} \mid \mathcal{P}(\mathbf{s}_{v_{1}}, \dots, \mathbf{s}_{v_{n}}), r_{\lt i}; \theta)
     + \sum_{j=1}^{L} \log P(s_{v_{n+1}}^{j} \mid \mathcal{P}(\mathbf{s}_{v_{1}}, \dots, \mathbf{s}_{v_{n}}), \boldsymbol{\tau}, s_{v_{n+1}}^{\lt j}; \theta) \Biggr)
```

同时优化推理 token 生成和目标物品 token 生成，权重相等。

---

### 阶段三：Reasoning Enhancement（推理强化，RL 阶段）

**挑战**：标准 GRPO 中的可验证奖励（Verifiable Pass Reward）在推荐场景面临**奖励稀疏问题**——大多数 rollout 无法命中目标物品，奖励全为 0，导致组优势（Group Advantage）消失，RL 失效。

**解决方案：Rollout-Beam 奖励**

用 beam search 在推理路径后生成 $`K`$ 个候选物品，取最佳匹配作为奖励：

```math
\mathcal{R}_{\text{Rollout-Beam}} = \max_{\hat{s}_{v_{n+1}} \in \mathcal{B}} \sum_{l=1}^{L} \mathbb{I}(\hat{s}_{v_{n+1}}^l = s_{v_{n+1}}^l)
```

其中 beam search 结果集：

```math
\mathcal{B} = \text{BeamSearch}\Bigl(P(\mathbf{s}_{v_{n+1}} \mid \mathbf{H}, \mathbf{\tau};\theta), K\Bigr)
```

**设计优势**：
- 利用用户偏好的**多有效性**（Multi-Validity）：用户可能喜欢多个合理的推荐结果，beam 候选中命中任意一个都是有效奖励信号
- 训练-推理一致性：奖励计算方式与 beam search 推理对齐
- 更密集的学习信号：通过多路径评估替代单点命中判断

**RL 训练细节**：
- 框架：VERL（用于分布式训练和可扩展奖励计算）
- 算法：GRPO，每次采样 $`|G|=16`$ 条 CoT 路径，每条路径 beam width $`K=32`$
- 超参：2 epochs，lr=1e-5，KL 系数 $`\beta=0.001`$，clip ratio $`\epsilon=0.2`$

## 工业部署："Think-Ahead" 架构

### 核心挑战

多步推理的计算开销与工业推荐系统的实时延迟要求（毫秒级）之间存在根本矛盾。

### 解决方案：两阶段解耦推理

**离线阶段（Stage 1）：推理引导的前缀生成**

对用户 $`u`$ 的历史 $`H_u`$，采样 $`T`$ 条多样化推理路径，并为每条路径进行 beam search 生成前两个 Itemic Token（前缀）：

```math
\mathcal{A}_u^{(i)} = \text{BeamSearch}\Bigl(P\bigl(\hat{s}_{v_{n+1}}^1,\hat{s}_{v_{n+1}}^2 \mid H_u,\mathbf{\tau}^{(i)};\theta\bigr), m\Bigr)
```

所有路径的前缀取并集，构成用户个性化候选空间：

```math
\mathcal{C}_u = \bigcup_{i=1}^T \mathcal{A}_u^{(i)}
```

将 $`T \times m`$ 个高潜力物品前缀缓存到分布式存储系统。

**在线阶段（Stage 2）：前缀约束的实时解码**

用户请求到来时，检索 $`\mathcal{C}_u`$，使用**实时更新的 OneRec 模型**在前缀约束下快速完成最后一个 token 的解码：

```math
\hat{\mathbf{s}}_{v_{n+1}} = \underset{\mathbf{s}_{v_{n+1}}}{\arg\max} \ P_{h_{\mathrm{online}}}\bigl(\mathbf{s}_{v_{n+1}} \bigm| \mathbf{s}_{v_1},\dots, \mathbf{s}_{v_n}\bigr)
\quad \text{s.t.} \quad (\hat{s}_{v_{n+1}}^1,\hat{s}_{v_{n+1}}^2) \in \mathcal{C}_u
```

**架构优势**：
- 计算密集的推理路径生成在离线完成，不影响在线延迟
- 在线阶段只需解码一个 token，极低延迟
- 利用实时更新的 OneRec 模型，保证时效性
- 推理结果物化为语义假设，有效指导在线解码

### 工业模型配置

- **Backbone**：Qwen-8B（初始化自公开预训练权重）
- **词表扩展**：新增 24,576 个 token（三级层次化 Itemic Token，每级 8,192 个）+ 2 个边界 token
- **每日增量训练**：80 张旗舰 GPU，每天处理约 200 亿 token

## Case Study 展示

论文展示了多种推理场景：

1. **情感感知对话推荐**：用户表达负面情绪时，模型检测情感信号，将推荐从通用内容转向轻松正向内容
2. **细粒度兴趣推理**：生成多样化推理路径，捕捉具体的游戏机制、叙事模式等细粒度偏好
3. **推理-推荐一致性验证**：在中间推理步骤应用 beam search，确认推理过程真正引导推荐生成（而非事后解释）
4. **Itemic-Textual 交织推理**：Itemic Token 提供精确内容锚定，文本 Token 提供因果表达，交织推理超越单一模态

## 核心贡献总结

1. **统一框架**：首次在单一模型中无缝集成对话、推理和个性化推荐，通过 Itemic Token 桥接离散推荐物品与连续推理空间

2. **Itemic Alignment**：四类多任务预训练 + Token Warm-up/Multi-Task Integration 两阶段策略，使 LLM 真正"理解"物品语义

3. **Reasoning Activation**：从精简上下文蒸馏推理轨迹，再迁移到嘈杂工业序列，解决工业场景 CoT 激活难题

4. **Rollout-Beam 奖励**：利用用户偏好多有效性设计 RL 奖励，解决推荐场景奖励稀疏问题，提供更密集的学习信号

5. **Think-Ahead 部署架构**：离线推理路径+前缀生成 + 在线前缀约束实时解码，在保证推理质量的同时满足工业延迟要求

---

## 局限与未来工作

- 公开数据集序列较短稀疏，Reasoning Activation 阶段使用了手工构建的类别 CoT 作为替代
- 未来可探索更高效的推理路径生成策略，进一步降低离线计算开销
- Think-Ahead 架构中前缀数量 $`T \times m`$ 的最优配置仍需深入研究
