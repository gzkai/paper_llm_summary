## 研究背景与动机

### 推荐基座模型的三大必要属性

作者认为一个理想的推荐基座模型必须同时具备：

| 属性 | 含义 |
|------|------|
| **Scalability（可扩展性）** | 随参数量和数据量增大，性能持续稳定提升 |
| **Extensibility（可延展性）** | 低成本无缝适配任意数量的现有场景或新增场景 |
| **Efficiency（高效性）** | 在多场景海量数据下，训练和推理的计算开销保持低廉 |

### 现有方法的三大痛点

现有多场景建模方法遵循"先对齐再分解（harmonize-then-decompose）"范式，存在以下根本性缺陷：

**① 严格输入对齐损害可延展性**

工业级推荐系统中不同场景的特征体系差异巨大。以美团外卖为例，商家推荐（HP）和菜品推荐（PHF）的特征 Schema 天然不兼容，强制对齐不仅极易出错，还会导致大量信息丢失。

**② 手工结构设计缺乏可扩展性**

现有方法依赖专家经验手工设计场景共享/特定模块（如 MoE），这类精巧结构未经 Scaling 验证，无法从数据规模增长中获益。

**③ 数据量激增带来高昂成本**

传统 DLRM 范式训练成本与数据量线性相关，多场景数据的暴力整合导致训练开销急剧攀升。

## 核心方法：MTFM 框架

MTFM 由三大模块组成：**多场景用户级样本聚合**、**异构 Tokenization + 混合目标注意力架构**、**训练与推理系统优化**。

---

### 1. 多场景用户级样本聚合

#### 1.1 训练阶段

将特定时间窗口内多个场景的所有曝光行为按**用户粒度**聚合，每个用户对应一个训练样本：

```math
X = [\{H_i\}_{i=1}^{N_H}, \{R_i\}_{i=1}^{N_R}, \{U^s\}_{s=1}^{N_S}, \{ \{C_i^s, I_i^s \}_{i=1}^{N_E^s} \}_{s=1}^{N_S} ]
```

```math
Y = \{ \{Y_i^s \}_{i=1}^{N_E^s} \}_{s=1}^{N_S}
```

其中 $` C_i^s, I_i^s, Y_i^s `$ 分别为场景 $` s `$ 第 $` i `$ 次曝光的交叉特征、物品特征和标签；$` H_i, R_i `$ 为跨场景共享的历史序列和实时序列。

**关键设计**：用户行为序列（$` H, R `$）跨所有场景共享，而用户属性、交叉特征、物品特征均为场景独立（scenario-specific）。这一设计源于场景间的客观差异——商家推荐场景天然缺少菜品 tag 特征，两个场景的 Feature Schema 无法直接对齐。

#### 1.2 推理阶段

每个场景部署对应的**子图（Subgraph）**，按请求粒度聚合候选物品，只计算与当前场景相关的组件，跳过其他场景的特定模块，避免冗余计算。

---

### 2. MTFM 模型架构

#### 2.1 异构 Tokenization

将输入特征表示为变长异构 Token 序列，共三类 Token：

**H-Token（历史行为 Token）**：历史序列中每个物品对应一个 H-Token，不同历史序列使用不同 MLP 作为 Tokenizer：

```math
\mathbf{h}_{ij} = \text{MLP}_i(\text{Emb}(h_{ij}))
```

所有历史序列的 Token 按时间顺序排列，形成 $` \mathbf{H} \in \mathbb{R}^{L_H \times d_{\text{model}}} `$。

**R-Token（实时行为 Token）**：类似地，实时序列 Token 化为 $` \mathbf{R} \in \mathbb{R}^{L_R \times d_{\text{model}}} `$。

**T-Token（目标物品 Token）**：每个曝光行为对应一个 T-Token，拼接用户画像、交叉特征和物品特征，使用场景专属 MLP 映射：

```math
\mathbf{t}_{i}^s = \text{MLP}_{s} (\text{Emb}(U^s) \| \text{Emb}(C_i^s) \| \text{Emb}(I_i^s))
```

**最终输入**：三类 Token 拼接形成初始输入矩阵：

```math
\mathbf{X}^{(0)} = (\mathbf{H}; \mathbf{R}; \mathbf{T}) \in \mathbb{R}^{N \times d_{\text{model}}}
```

其中 $` N = L_H + L_R + L_T `$ 为总 Token 数，随用户不同而变化。

#### 2.2 混合目标注意力架构（Hybrid Target Attention, HTA）

**设计动机**：多场景场景下 Token 序列长度大幅增加，标准 Transformer 的 $` O(N^2) `$ 复杂度不可接受。HTA 借鉴 LLM 中的混合注意力思路，以 Full Attention 与 Target Attention 交错的方式，在建模能力与计算效率之间取得最优平衡。

**整体结构**：$` B `$ 个 Block 堆叠，每个 Block 包含 1 层 Full Attention Layer + $` K `$ 层 Target Attention Layer。

##### Full Attention Layer

首先对输入 Token 施加**分组 LayerNorm（GLN）**：不同历史序列的 H-Token、不同实时序列的 R-Token、不同场景的 T-Token 分别属于不同组，以适应各序列特征的多样分布：

```math
\widetilde{\mathbf{X}}^{(l)} = \text{GLN}(\mathbf{X}^{(l)})
```

采用 **GQA（Grouped-Query Attention）** 降低显存开销，$` H `$ 个 Query Head 共享 $` G `$ 个 KV Head（$` r = H/G `$）：

```math
\mathbf{U}^{(l)}, \{\mathbf{Q}^{(l,h)}\}_{h=1}^{H}, \{\mathbf{K}^{(l,g)}, \mathbf{V}^{(l,g)} \}_{g=1}^G = \text{Split}(\phi_1(f_1^{(l)}(\widetilde{\mathbf{X}}^{(l)})))
```

```math
\textbf{A}^{(l,h)} = \phi_2(\mathbf{Q}^{(l,h)} \mathbf{K}^{(l,g)T} \odot \mathbf{M}) \mathbf{V}^{(l,g)}, \quad g = \lceil h/r \rceil
```

```math
\mathbf{X}^{(l+1)} = f_2^{(l)}(\text{GLN}(\mathbf{A}^{(l)}) \odot \mathbf{U}^{(l)}) + \mathbf{X}^{(l)}
```

**动态 Mask 矩阵** $` \mathbf{M} `$ 按以下规则构建，防止信息泄露：
1. H-Token 对所有 Token 可见
2. R-Token 仅对时间戳更晚的 Token 可见
3. T-Token 仅对自身可见

##### Target Attention Layer

仅更新 T-Token 的 Embedding，其余 Token 通过 Shortcut 连接直接传递：

```math
\widetilde{\mathbf{X}}^{(l)}_T = \widetilde{\mathbf{X}}^{(l)}[L_H+L_R:], \quad \mathbf{M}_T = \mathbf{M}[L_H+L_R:]
```

```math
\mathbf{U}^{(l)}_T, \{\mathbf{Q}^{(l,h)}_T\}_{h=1}^H = \text{Split}(\phi_1(f_{uq}^{(l)}(\widetilde{\mathbf{X}}_T^{(l)})))
```

```math
\{\mathbf{K}^{(l,g)}, \mathbf{V}^{(l,g)}\}_{g=1}^G = \text{Split}(\phi_1(f_{kv}^{(l)}(\widetilde{\mathbf{X}}^{(l)})))
```

```math
\textbf{A}^{(l,h)}_T = \phi_2(\mathbf{Q}_T^{(l,h)} \mathbf{K}^{(l,g)T} \odot \mathbf{M}_T) \mathbf{V}^{(l,g)}, \quad g = \lceil h/r \rceil
```

```math
\mathbf{X}^{(l+1)} = (\mathbf{X}^{(l)}[:L_H+L_R]; \mathbf{X}^{(l+1)}_T)
```

**复杂度分析**：HTA 将计算复杂度从 $` O(N^2) `$ 降低至 $` O\left(\frac{KNL_T + N^2}{K+1}\right) `$，由于 $` L_T \ll N `$，效果显著。

**最终输出**：最后一层 T-Token Embedding 输入 **MMoE** 模块，计算多场景多目标的预估分。

### 3. 训练与推理系统优化

#### 3.1 数据 Pipeline

将特征分为**全场景共享**（$` H, R `$）和**场景独立**（$` U, C, I `$）两部分：
- 场景独立特征先在各场景内部按用户粒度聚合，再列拼接形成统一特征表示
- 最后与全场景共享特征按用户粒度合并
- 避免 Pipeline 重复计算和冗余存储，提升系统鲁棒性和资源利用效率

#### 3.2 训练优化

**CPU-GPU Pipeline 优化**：
- 推荐系统中大量特征存于 CPU，而 LLM 中所有特征都在 GPU，这导致严重的 CPU-GPU 同步阻塞
- 通过 CUDA Profiler 系统性定位阻塞点，消除同步点实现 CPU-GPU 执行 Overlap
- 将多次 D2D 拷贝操作合并为单次原子操作
- 训练吞吐提升 **20%**

**Custom Kernel 开发**：
- **增强版 FlashAttention-2**：针对动态 Mask 的不连续内存访问，设计连续对齐的 Mask 内存布局，利用共享内存流水线隐藏 Mask 加载延迟
- **Triton 融合算子**：将 GLN 和动态 Mask 构建融合，消除中间结果反复读写，利用向量化内存访问提升带宽利用率
- 以上 Kernel 优化额外带来 **57%** 吞吐提升

#### 3.3 推理优化

**结构化稀疏（Structured Sparsity）**：利用 NVIDIA Ampere 架构的稀疏张量核心，在 HSTU 的 UVQK 和输出层实施 2:4 结构化剪枝，显存降低 50%，吞吐提升约 **10%**，延迟降低约 **0.2ms**。

**细粒度注意力剪枝**：针对动态 Mask 中的不规则稀疏性，动态剔除 Padding 产生的无效计算，并屏蔽用户特征对目标特征的无效注意力权重，吞吐提升约 **5%**。

**场景化子图部署**：将完整计算图拆解为场景专属子图，消除本场景外的冗余计算；结合 BF16 半精度推理和 M-Falcon 智能批处理算法进一步提升吞吐。

### RQ3：Scaling Law 验证

**模型规模 Scaling**：在 10× 到 70× 参数规模范围内，MTFM 在 SQS/HP/PHF 三个场景的 CTCVR GAUC 均呈现稳定的 Scaling Law（$` R^2 > 0.93 `$），斜率一致。

**数据规模 Scaling**：随训练 Token 数增加，所有模型尺寸的性能持续提升；大模型与小模型之间的性能差距随数据量增大而逐渐扩大，表明 MTFM 能有效将数据规模转化为性能增益。

### RQ4：注意力可解释性分析

注意力热力图可视化显示：
- 所有场景的 T-Token 均能有效聚合来自整个多场景空间的 H-Token 信息
- 不同场景呈现差异化的学习模式：HP 和 SQS 的 T-Token 对自身 H-Token 的注意力权重更高；PHF 的 T-Token 则更集中于 PHF 历史，同时 HP 和 SQS 历史也起辅助作用
- 热力图呈现清晰的块状注意力模式，证明 MTFM 具有良好的场景感知能力

## 核心贡献总结

1. **统一基座架构**：MTFM 首次将多场景工业推荐统一到单一 Transformer 框架，通过异构 Tokenization 实现无需输入对齐的跨场景建模，同时满足 Scalability、Extensibility、Efficiency 三大属性

2. **混合目标注意力（HTA）**：Full Attention（全局依赖捕获）与 Target Attention（高效目标更新）交错设计，配合 GQA，将复杂度从 $` O(N^2) `$ 降至 $` O\left(\frac{KNL_T + N^2}{K+1}\right) `$，训练吞吐 2× 提升，性能几乎无损

3. **系统-模型协同设计**：CPU-GPU Pipeline 优化（+20%）+ Custom Kernel（+57%）+ 结构化稀疏（+10%）+ 场景化子图部署，实现训练/推理吞吐近翻倍

4. **工业级验证**：美团外卖三大场景（HP/SQS/PHF）全面验证，消除多场景"跷跷板效应"，在线 A/B 测试订单量显著提升，且延迟更低
