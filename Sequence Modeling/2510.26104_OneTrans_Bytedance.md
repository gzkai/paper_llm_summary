## 研究背景与动机

### 工业推荐排序的两大模块

工业推荐排序系统（DLRM 范式）通常包含两个独立模块：

1. **序列建模（Sequence Modeling）**：将用户多行为序列编码为候选感知表示（DIN、BST、LONGER 等）
2. **特征交叉（Feature Interaction）**：学习非序列特征（用户画像、商品画像、上下文）的高阶交叉（DCN、Wukong、RankMixer 等）

传统架构为 **encode-then-interaction 流水线**：先将行为序列压缩为固定向量，再与非序列特征拼接后送入特征交叉模块。

### 现有方法的两大局限

| 局限性 | 具体问题 |
|--------|---------|
| **信息流受限** | encode-then-interaction 限制了双向信息流，静态/上下文特征无法有效塑造序列表示 |
| **执行碎片化** | 两个独立模块无法统一优化，也无法复用 LLM 工程优化（KV 缓存、FlashAttention、混合精度） |

### 核心洞见

将序列建模与特征交叉统一到**单一 Transformer 骨干**中，使 RecSys 的 Scaling 与 LLM 实践对齐，同时继承成熟的 LLM 工程优化。

## 核心方法：OneTrans 架构

### 总体结构

```math
\hat{y}_{u,i} = f\left(i \,\big|\, \mathcal{NS}, \mathcal{S}; \Theta\right)
```

其中 $`\mathcal{NS}`$ 为非序列特征集合，$`\mathcal{S}`$ 为用户历史行为序列集合。

**初始 token 序列**：

```math
\mathbf{X}^{(0)} = \big[ \text{S-tokens} \,;\, \text{NS-tokens} \big] \in \mathbb{R}^{(L_{\text{S}} + L_{\text{NS}}) \times d}
```

每个 OneTrans Block 的更新规则（Pre-Norm Transformer）：

```math
\mathbf{Z}^{(n)} = \mathrm{MixedMHA}\!\left(\mathrm{Norm}\big(\mathbf{X}^{(n-1)}\big)\right) + \mathbf{X}^{(n-1)}
```

```math
\mathbf{X}^{(n)} = \mathrm{MixedFFN}\!\left(\mathrm{Norm}\big(\mathbf{Z}^{(n)}\big)\right) + \mathbf{Z}^{(n)}
```

---

### 一、统一 Tokenizer

#### 非序列特征 Tokenization（NS-tokens）

工业系统通常有数百个特征，提供两种方案：

**方案 A：Group-wise Tokenizer**（对齐 RankMixer）
- 人工将特征分组，每组拼接后过独立 MLP：

```math
\text{NS-tokens} = \big[\,\text{MLP}_1(\text{concat}(\mathbf{g}_1)),\dots, \text{MLP}_{L_{NS}}(\text{concat}(\mathbf{g}_{L_{NS}}))\,\big]
```

**方案 B：Auto-Split Tokenizer**（默认，实验更优）
- 所有特征一次性拼接投影，再均匀切分：

```math
\text{NS-tokens} = \text{split}\Big(\text{MLP}(\text{concat}(\mathcal{NS})),\,L_{NS}\Big)
```

> 消融实验表明 Auto-Split 优于 Group-wise（CTR AUC +0.10%，UAUC +0.30%），且减少 Kernel 启动开销。

#### 序列特征 Tokenization（S-tokens）

接受多行为序列 $`\mathcal{S} = \{ \mathbf{S}_1, \dots, \mathbf{S}_n \}`$，每个序列独立投影到统一维度 $`d`$：

```math
\tilde{\mathbf{S}}_i = \big[\, \mathrm{MLP}_i(\mathbf{e}_{i1}), \dots, \mathrm{MLP}_i(\mathbf{e}_{iL_i}) \,\big] \in \mathbb{R}^{L_i \times d}
```

多序列合并策略（两种）：
1. **时间感知（Timestamp-aware）**：按时间戳交叉排列，加序列类型指示符 → **实验更优**
2. **意图排序（Timestamp-agnostic）**：按用户意图强度排列（购买 → 加购 → 点击），序列间插入可学习 `[SEP]` token

最终：

```math
\text{S-Tokens} = \mathrm{Merge}\big(\tilde{\mathbf{S}}_1, \dots, \tilde{\mathbf{S}}_n\big) \in \mathbb{R}^{L_S \times d}, \quad L_S = \sum_{i=1}^n L_i + L_\text{SEP}
```

### 二、混合参数化（Mixed Parameterization）

**核心设计**：S-tokens 共享参数，NS-tokens 使用 token 独立参数。

**混合 Causal Attention**：

```math
\big(\mathbf{q}_i, \mathbf{k}_i, \mathbf{v}_i\big) = \big(\mathbf{W}^{Q}_i \mathbf{x}_i, \;\mathbf{W}^{K}_i \mathbf{x}_i, \; \mathbf{W}^{V}_i \mathbf{x}_i \big)
```

其中参数矩阵遵循混合策略：

```math
\mathbf{W}^{\Psi}_i =
\begin{cases}
\mathbf{W}^{\Psi}_{\mathrm{S}}, & i \le L_S \quad (\text{S-tokens 共享}) \\[1.5mm]
\mathbf{W}^{\Psi}_{\mathrm{NS},i}, & i > L_S \quad (\text{NS-tokens 独立})
\end{cases}
```

**混合 FFN**：

```math
\mathrm{MixedFFN}(\mathbf{x}_i) = \mathbf{W}^{2}_i \, \phi(\mathbf{W}^{1}_i \mathbf{x}_i)
```

同样遵循上述混合参数化策略。

**因果掩码（Causal Mask）的三重作用**：
1. **S 侧**：每个行为 token 只关注其历史，时间感知序列自然因果；意图排序序列中高意图信号可过滤低意图行为
2. **NS 侧**：每个 NS-token 关注**全部** S 历史（类似 target attention 聚合），以及前序 NS-tokens
3. **支持金字塔剪枝**：因果掩码将信息逐渐集中到靠后位置，天然支持逐层剪枝

**设计动机**：
- S-tokens 来自同质行为序列，共享参数合理且高效
- NS-tokens 来自异构特征源（用户 ID、商品类目、价格等），语义差异大，需独立参数以保留区分性
- 使用 **RMSNorm Pre-Norm**（而非 Post-Norm）防止异构 token 统计差异导致的注意力崩溃

---

### 三、Pyramid Stack（金字塔堆叠）

**动机**：因果掩码使信息集中于靠后位置，早期 S-tokens 的信息已被后续 token 聚合，可以安全剪枝。

**操作**：每层仅对最近 $`L'`$ 个 S-token 计算 Query，Keys/Values 仍覆盖全序列：

```math
\mathbf{q}_i = \mathbf{W}^{Q}_i\,\mathbf{x}_i, \qquad i \in \mathcal{Q} = \{L{-}L'{+}1,\dots,L\}
```

注意力后只保留 $`\mathcal{Q}`$ 中位置的输出，逐层缩减 token 数量直至与 NS-token 数量相当。

**两大收益**：
1. **渐进蒸馏**：长行为历史逐层压缩到少量尾部 Query，容量集中在最有信息量的事件上
2. **计算效率**：注意力成本降为 $`O(LL'd)`$，FFN 与 $`L'`$ 线性相关；在相同 TFLOPs 预算下支持约 **1.75× 更长的序列**

**实际配置**：
- OneTrans-S：序列从 1190 个 token 线性缩减至 12 个
- OneTrans-L：序列从 1500 个 token 线性缩减至 16 个（每层对齐到 32 的倍数）

### 四、训练与部署优化

#### 跨请求 KV 缓存（Cross-Request KV Caching）

工业推荐场景中，同一请求的所有候选商品共享相同的用户行为序列（S-tokens），只有 NS-tokens 因候选不同而变化。

**两阶段范式**：
- **Stage I（S 侧，每请求一次）**：处理所有 S-tokens，缓存 Key/Value 对和注意力输出
- **Stage II（NS 侧，每候选一次）**：仅计算 NS-tokens 并与缓存的 S 侧 Key/Value 做交叉注意力

将时间复杂度从 $`O(C)`$ 降至 $`O(1)`$（$`C`$ 为候选数量）。

**跨请求扩展**：用户行为序列是追加式的，新请求复用上次缓存，仅计算新增行为的增量 Key/Value，将序列计算从 $`O(L)`$ 降至 $`O(\Delta L)`$。

#### 统一 LLM 工程优化

- **FlashAttention-2**：Tiling + Kernel 融合，降低注意力 I/O 和激活内存
- **混合精度训练（BF16/FP16）+ 激活重计算**：丢弃部分前向激活并在反向传播时重算，大幅节省显存
- **半精度推理**：推理端到端使用低精度，显著降低延迟

### RQ2：消融实验

以 OneTrans-S 为参考：

| 类型 | 变体 | CTR AUC↑ | CTR UAUC↑ | CVR AUC↑ | CVR UAUC↑ |
|------|------|----------|-----------|----------|-----------|
| **输入** | Group-wise Tokenizer | −0.10% | −0.30% | −0.12% | −0.10% |
| | Timestamp-agnostic Fusion | −0.09% | −0.22% | −0.20% | −0.21% |
| | Timestamp-agnostic w/o SEP tokens | −0.13% | −0.32% | −0.29% | −0.33% |
| **Block** | 所有 token 共享参数 | −0.15% | −0.29% | −0.14% | −0.29% |
| | Full Attention（替代 Causal） | +0.00% | +0.01% | −0.03% | +0.06% |
| | 不使用 Pyramid Stack | −0.05% | +0.06% | −0.04% | −0.42% |

**设计洞见**：
1. **Auto-Split Tokenizer** 优于人工分组，让模型自动学习非序列 token 的构建方式更有效
2. **时间感知融合** 在有时间戳时优于意图排序，时序信息优先于事件重要性
3. **[SEP] token** 在无时间戳时帮助模型区分不同序列边界
4. **NS-token 独立参数** 显著优于共享参数，保留异构特征的区分性
5. **因果注意力与全注意力** 性能相近，但全注意力禁用 KV 缓存，工程代价高
6. **Pyramid Stack** 在固定 FLOPs 预算下支持 1.75× 更长序列，且性能无损

### RQ4：Scaling Law 验证

沿三个轴探索 Scaling：
1. **长度（Length）**：输入 token 序列长度 → **收益最大**
2. **深度（Depth）**：堆叠层数 → 收益大于宽度，但增加串行计算
3. **宽度（Width）**：隐层维度 → 更易并行化，但收益略小于深度

**OneTrans vs RankMixer Scaling 曲线**（$`\Delta`$UAUC vs log FLOPs）：
- 两者均呈近似 **log-linear** 趋势
- OneTrans 斜率**更陡**：统一骨干的 Scaling 效率高于 RankMixer 的 MoE-centric 扩展（后者主要扩展 FFN 隐层维度）

## 核心贡献总结

1. **统一框架**：提出 OneTrans，用单一 Transformer 骨干替代 encode-then-interaction 流水线，通过统一 Tokenizer 将序列特征和非序列特征编码为统一 token 序列，在单一计算图中同时完成序列建模和特征交叉

2. **推荐场景定制**：混合参数化策略——同质 S-tokens 共享参数，异质 NS-tokens 使用 token 独立参数，在参数效率与表达能力之间取得平衡

3. **高效训练与部署**：Pyramid Stack（逐层剪枝序列 Query）+ Cross-Request KV Caching（跨候选/跨请求复用计算）+ FlashAttention + 混合精度，使 330M 参数的 OneTrans-L 推理延迟与 10M 参数的 DCNv2+DIN 持平

4. **Scaling Law 验证**：在 29.1B 工业数据上验证 OneTrans 具有更陡的 log-linear Scaling 曲线，参数与计算效率优于 RankMixer 的 MoE-centric 扩展

5. **全量部署验证**：在字节跳动电商 Feeds 场景 GMV/u 提升 **+5.68%**，Mall 全场景 GMV/u 提升 **+3.67%**，同时延迟降低约 3-4%

## 与相关工作的关键区别

| 方法 | 类型 | 局限性 |
|------|------|--------|
| RankMixer | 特征交叉 | 无序列建模，encode-then-interaction |
| LONGER | 序列建模 | 无特征交叉，encode-then-interaction |
| HiFormer | 特征交叉 | 异构特征交叉，但仍与序列模块分离 |
| InterFormer | 桥接尝试 | 仍维持两个独立模块，跨模块架构复杂 |
| **OneTrans** | **统一** | **单一骨干，联合建模，直接继承 LLM 优化** |
