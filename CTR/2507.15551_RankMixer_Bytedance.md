## 研究背景与动机

### 工业推荐系统的 Scaling 困境

LLM 的成功激发了推荐系统 Scaling 的需求，但工业级推荐系统面临两大独特挑战：

1. **严苛的延迟与 QPS 约束**：工业推荐系统必须满足毫秒级延迟和极高的 QPS 要求，不能简单地堆叠参数
2. **CPU 时代遗留的低效架构**：现有排序模型（DCN、DHEN、Wukong 等）的特征交叉模块是在 CPU 时代设计的，其核心算子在现代 GPU 上是 **Memory-bound** 而非 Compute-bound，导致 MFU（Model FLOPs Utilization）极低（个位数百分比），Scaling 收益难以实现

### 现有方法的局限

| 方法类型 | 局限性 |
|---------|--------|
| 简单扩宽/加深 DLRM | 收益有限甚至为负 |
| DCN / RDCN / AutoInt | 参数量与计算量失衡，FLOPs 增长过快 |
| DHEN / Wukong | Scaling 曲线不理想，计算代价过高 |
| 标准 Transformer Self-Attention | 异构特征空间内积相似度难以学习，内存开销大，MFU 低 |
| 标准 MoE | 专家不均衡、欠训练问题，Scaling 效果次优 |

### 核心洞见

**硬件对齐的模型设计哲学**：推荐系统的 Scaling 必须同时满足：
- 架构与现代 GPU 硬件对齐，最大化 MFU 和计算吞吐量
- 模型设计充分利用推荐数据的特性（异构特征空间、数百个特征域的个性化交叉）

## 核心方法：RankMixer 架构

### 总体结构

RankMixer 由 $`T`$ 个输入 token 经过 $`L`$ 个 RankMixer Block 处理，最后通过 Mean Pooling 输出。每个 RankMixer Block 包含两个核心组件：

```math
\mathbf{S}_{n-1} = \mathrm{LN}\left(\mathrm{TokenMixing}\left(\mathbf{X}_{n-1}\right) + \mathbf{X}_{n-1}\right)
```

```math
\mathbf{X}_{n} = \mathrm{LN}\left(\mathrm{PFFN}\left(\mathbf{S}_{n-1}\right) + \mathbf{S}_{n-1}\right)
```

其中 $`\mathbf{X}_{n} \in \mathbb{R}^{T \times D}`$ 为第 $`n`$ 层输出，$`D`$ 为隐藏维度。

---

### 一、输入层与特征 Tokenization

**问题**：推荐模型通常有数百个特征，维度各异。
- 每个特征一个 token → token 数过多，每个 token 分配的参数和计算量碎片化，GPU 核利用率低
- 单个 token → 退化为简单 DNN，无法区分不同特征空间，高频特征淹没低频特征

**解决方案：基于语义的分组 Tokenization**

利用领域知识将语义相近的特征分组，拼接后按固定维度切分为 $`T`$ 个 feature token：

```math
x_i = \mathrm{Proj}\!\left(e_{\text{input}}\bigl[d\cdot(i-1) : d\cdot i\bigr]\right), \quad i = 1,\dots,T
```

每个 token $`x_i \in \mathbb{R}^{D}`$ 代表一组语义相近的特征嵌入。

---

### 二、Multi-head Token Mixing（多头 Token 混合）

**设计动机**：Self-Attention 在推荐场景中次优——推荐特征空间天然异构，用内积计算用户 ID 与视频 ID 之间的相似度在语义上不合理，且计算和内存开销大。

**操作原理**：

将每个 token $`\mathbf{x}_t`$ 均匀切分为 $`H`$ 个 head：

```math
\left[\mathbf{x}_t^{(1)} \;\Vert\; \mathbf{x}_t^{(2)} \;\Vert\;\dots\;\Vert\; \mathbf{x}_t^{(H)}\right] = \mathrm{SplitHead}\left(\mathbf{x}_t\right)
```

对第 $`h`$ 个 head，将所有 token 的同一 head 拼接成新 token：

```math
\mathbf{s}^{h} = \mathrm{Concat}\left(\mathbf{x}_1^{h}, \mathbf{x}_2^{h}, \dots, \mathbf{x}_T^{h}\right)
```

输出 $`\mathbf{S} \in \mathbb{R}^{H \times \frac{TD}{H}}`$，设 $`H=T`$ 保持 token 数不变以支持残差连接。

**关键特性**：
- **无参数操作**：Token Mixing 本身不引入额外参数，仅是特征维度的重排（Shuffle）
- **计算高效**：相比 Self-Attention，避免了 $`O(T^2)`$ 的注意力权重矩阵，消除 Memory-bound 瓶颈
- **跨特征空间交互**：不同语义子空间的特征通过 head 维度实现信息交换

---

### 三、Per-token FFN（逐 Token 独立 FFN）

**设计动机**：传统 DLRM/DHEN 将所有特征空间的特征混合在同一个交互模块中，高频特征容易主导并淹没低频/长尾信号。

**操作原理**：为每个 token 分配独立的 FFN 参数，隔离不同特征子空间的建模：

```math
\mathbf{v}_t = f_{\text{pffn}}^{t,2}\left(\mathrm{Gelu}\left(f_{\text{pffn}}^{t,1}\left(\mathbf{s}_t\right)\right)\right)
```

```math
f_{\text{pffn}}^{t,i}(\mathbf{x}) = \mathbf{x}\mathbf{W}_{\text{pffn}}^{t,i} + \mathbf{b}_{\text{pffn}}^{t,i}
```

其中 $`\mathbf{W}_{\text{pffn}}^{t,1} \in \mathbb{R}^{D \times kD}`$，$`k`$ 为调节隐藏维度的超参数。

**与其他方法的区别**：
- vs. **共享 FFN（Transformer）**：不同 token 使用独立参数，避免参数共享导致的特征空间干扰
- vs. **MMoE**：MMoE 所有专家接收相同输入，Per-token FFN 每个 FFN 接收不同 token 输入；RankMixer 同时切分输入和参数
- **效率优势**：参数量增加 $`T`$ 倍，但 FLOPs 不变（并行计算，GPU Tensor Core 充分利用）

---

### 四、Sparse MoE in RankMixer

**标准 Sparse-MoE 在推荐场景的两大问题**：
1. **均匀 Top-k 路由**：对所有 token 一视同仁，信息量高的 token 得不到足够专家，信息量低的 token 浪费专家预算
2. **专家欠训练**：Per-token FFN 已将参数乘以 token 数，再加非共享专家导致专家数量爆炸，路由极度不均衡，专家严重欠训练

**两大解决策略**：

#### 1. ReLU Routing（动态稀疏路由）

用 ReLU 门控 + 自适应 $`\ell_1`$ 惩罚替代标准 Top-k + Softmax：

```math
G_{i,j} = \mathrm{ReLU}\bigl(h(\mathbf{s}_i)\bigr), \quad \mathbf{v}_i = \sum_{j=1}^{N_e} G_{i,j}\, e_{i,j}(\mathbf{s}_i)
```

稀疏度由正则项控制：

```math
\mathcal{L} = \mathcal{L}_{\text{task}} + \lambda\,\mathcal{L}_{\mathrm{reg}}, \quad \mathcal{L}_{\mathrm{reg}} = \sum_{i=1}^{N_t}\sum_{j=1}^{N_e} G_{i,j}
```

**效果**：高信息量 token 动态激活更多专家，低信息量 token 激活更少，自适应分配专家预算。

#### 2. DTSI-MoE（Dense-Training Sparse-Inference）

训练时使用密集路由器 $`h_{\text{train}}`$（不施加稀疏惩罚），推理时使用稀疏路由器 $`h_{\text{infer}}`$（施加 $`\mathcal{L}_{\mathrm{reg}}`$），两个路由器同时更新但推理只用后者。

**效果**：训练时所有专家充分获得梯度更新（避免"死亡专家"），推理时稀疏激活降低计算成本。

---

### 五、Scaling 方向分析

RankMixer 支持四个正交的 Scaling 轴：Token 数 $`T`$、模型宽度 $`D`$、层数 $`L`$、专家数 $`E`$。

**Dense 版本的参数量与 FLOPs**：

```math
\#\text{Param} \approx 2kLTD^2, \qquad \text{FLOPs} \approx 4kLTD^2
```

**实验结论**：与 LLM Scaling Law 一致——模型质量主要由总参数量决定，不同 Scaling 方向（深度 $`L`$、宽度 $`D`$、Token 数 $`T`$）效果几乎相同。从计算效率角度，更大的隐藏维度 $`D`$ 产生更大的矩阵乘法形状，MFU 更高。

**最终配置**：
- RankMixer-100M：$`D=768, T=16, L=2`$
- RankMixer-1B：$`D=1536, T=32, L=2`$

## 实验结果

### 数据集与评估设置

- **数据集**：抖音推荐系统在线日志，覆盖 300+ 特征，数十亿用户 ID，数亿视频 ID，每日万亿级记录，实验使用两周数据
- **评估指标**：Finish AUC/UAUC、Skip AUC/UAUC（AUC 提升 0.0001 即为显著改进）
- **Baseline**：DLRM-MLP、DCNv2、RDCN、MoE、AutoInt、HiFormer、DHEN、Wukong

---

| AutoInt | +0.10% | +0.14% | +0.12% | +0.23% | 19.2M | 307G |

---

### Scaling Law 对比

- **RankMixer 的 Scaling 曲线最陡**：在参数量和 FLOPs 两个维度上均优于所有对比方法
- Wukong 参数曲线较陡，但 FLOPs 增长更快，FLOPs 维度上差距更大
- DHEN Scaling 不理想，反映其交叉结构的扩展性限制
- 标准 MoE 因专家不均衡问题，Scaling 效果次优

---

### 消融实验

**组件消融（RankMixer-100M）**：

| 消融设置 | $`\Delta`$AUC |
|---------|-------------|
| 去除 skip connections | −0.07% |
| **去除 Multi-head Token Mixing** | **−0.50%** |
| 去除 Layer Normalization | −0.05% |
| **Per-token FFN → 共享 FFN** | **−0.31%** |

**Token 路由策略对比**：

| 路由策略 | $`\Delta`$AUC | $`\Delta`$Params | $`\Delta`$FLOPs |
|---------|-------------|----------------|----------------|
| All-Concat-MLP | −0.18% | 0.0% | 0.0% |
| All-Share | −0.25% | 0.0% | 0.0% |
| Self-Attention | −0.03% | +16% | **+71.8%** |
| **Multi-head Token Mixing** | **最优** | 0.0% | 0.0% |

> Self-Attention 性能略差于 Multi-head Token Mixing，且计算成本高出 71.8%，验证了在异构特征空间中内积相似度难以学习的判断。

---

### Sparse MoE 可扩展性

- **DTSI + ReLU 路由**：在激活比例降至 1/8 时，AUC 几乎无损，推理吞吐量提升 **+50%**
- **标准 Sparse MoE**：随激活专家数减少，性能单调下降
- **加负载均衡 Loss**：相比标准 MoE 有改善，但仍不及 DTSI + ReLU（根本原因在于专家训练不足而非路由器问题）
- **专家激活多样性**：ReLU 路由使不同 token 的激活比例自适应变化，高信息量 token 激活更多专家

---

### 在线部署成本分析

| 指标 | OnlineBase-16M | RankMixer-1B | 变化 |
|------|---------------|-------------|------|
| 参数量 | 15.8M | 1.1B | **↑ 70×** |
| FLOPs | 107G | 2106G | ↑ 20.7× |
| FLOPs/Param (G/M) | 6.8 | 1.9 | **↓ 3.6×** |
| **MFU** | 4.47% | **44.57%** | **↑ 10×** |
| 硬件精度 | fp32 | fp16 | ↑ 2× |
| **推理延迟** | 14.5ms | **14.3ms** | **持平** |

**延迟分析公式**：

```math
\text{Latency} = \frac{\#\text{Param} \times \text{FLOPs/Param ratio}}{\text{MFU} \times \text{Theoretical Hardware FLOPs}}
```

70× 参数增长被三重优化抵消：
1. **FLOPs/Param 降低 3.6×**：硬件对齐设计，70× 参数只带来 20× FLOPs
2. **MFU 提升 10×**：大矩阵乘法形状 + 并行 Per-token FFN 融合为单一 Kernel，从 Memory-bound 转为 Compute-bound
3. **量化提升 2×**：fp16 推理，理论峰值算力翻倍

## 核心贡献总结

1. **硬件对齐的统一架构**：提出 RankMixer，以 Multi-head Token Mixing + Per-token FFN 为核心，替代 CPU 时代遗留的异构手工特征交叉模块，将 MFU 从 4.5% 提升至 45%

2. **参数与计算解耦**：通过 Per-token FFN 的参数隔离设计，实现参数量增长与 FLOPs 增长的解耦（70× 参数 vs 20× FLOPs），为大规模 Scaling 提供工程可行性

3. **工业级 Sparse MoE**：提出 DTSI（Dense-Training Sparse-Inference）+ ReLU Routing 组合策略，解决推荐场景中专家欠训练和不均衡问题，在参数容量扩大 8× 以上时 AUC 几乎无损，推理吞吐提升 50%

4. **Scaling Law 验证**：在万亿级工业数据集上验证 RankMixer 具有最陡的 Scaling 曲线，与 LLM Scaling Law 规律一致（总参数量是性能的主要决定因素，不同 Scaling 方向效果相近）

5. **全量部署验证**：RankMixer-1B 在抖音 Feed 推荐全量上线，Active Day +0.3%，App Duration +1.08%；广告场景 AUC +0.73%，ADVV +3.90%
