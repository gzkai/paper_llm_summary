## 一、研究背景与动机

### 问题背景

Transformer 架构在 LLM 领域取得了巨大成功，但将其直接应用于工业级推荐系统的排序模型仍处于初期阶段。传统排序模型（DLRM）依赖专门设计的网络处理特征交互（如 DIN 处理序列特征、DeepFM/DCN 处理异构特征交互），这类架构：

- 主要依赖低计算量操作（向量运算、低维矩阵乘法），难以充分利用现代加速器的并行计算能力
- 无法捕捉高动态数据模式
- 缺乏可扩展性，模型容量受限

### 核心挑战：高特征稀疏性 × 低标签密度

将 Transformer 应用于排序模型面临两大固有挑战：

| 挑战 | 语言模型 | 推荐排序模型 |
|------|---------|------------|
| **特征稀疏性** | 低（BPE/WordPiece 将词表控制在数十万以内） | 极高（物品词表达十亿级，需要巨大的 Embedding 表） |
| **标签密度** | 高（每个位置都有 next-token 预测信号） | 低（仅目标物品有稀疏二元信号，如点击/购买） |

这种信息不对称（巨大参数空间仅由稀疏二元标签正则化）导致：
1. **严重过拟合**
2. **长历史序列上的大量计算浪费**

## 二、SORT 方法详解

SORT（**S**ystematically **O**ptimized **R**anking **T**ransformer）在标准 Transformer 基础上进行了 7 项系统性优化，分为两类：

### 2.1 解决高稀疏性与低标签密度

#### (1) Request-Centric 样本组织

传统方式（Impression-Centric）：每个候选物品独立构成一个样本，导致 Request 级不变特征（用户画像、历史序列）被重复处理 N 次。

SORT 采用 **Request-Centric** 组织方式，将一次请求的所有候选物品封装为一个统一样本：

$$\mathcal{S} = \langle \mathcal{H}, \mathcal{U}, \mathcal{C} \rangle$$

其中：
- $\mathcal{H} = (i_1, i_2, \dots, i_L)$：用户历史行为序列
- $\mathcal{U}$：用户画像（静态元数据 + 统计特征）
- $\mathcal{C} = \{c_j\}_{j=1}^N$：候选物品集合

优点：消除冗余计算，同时在一个样本内处理多个候选项，摊薄训练成本。

#### (2) 生成式预训练（Generative Pre-training）

采用 GPSD 框架的范式：
1. 基于用户点击序列训练一个 **next-item 预测模型**
2. 将学到的稀疏参数（物品 Embedding 表）迁移到排序模型
3. 在排序模型训练期间，**冻结**该 Embedding 表

实验结果对比：

| 策略 | CTR-AUC | CVR-AUC | AddCart-AUC |
|------|---------|---------|-------------|
| Transformer（从头训练） | 0.7175 | 0.8959 | 0.8455 |
| +sparse transfer（仅迁移） | 0.7162（-0.13pt） | 0.8943（-0.16pt） | 0.8449（-0.06pt） |
| +sparse transfer + sparse freeze | **0.7456（+2.81pt）** | **0.9093（+1.34pt）** | **0.8663（+2.08pt）** |

关键发现：仅迁移不冻结甚至会轻微降低性能；**冻结是关键**，带来高达 +2.81pt 的 CTR-AUC 提升。此外，冻结后支持多 epoch 训练而不过拟合。

### 2.2 Transformer 架构优化

#### (3) 特殊 Token（Special Tokens）

在输入序列中引入两类特殊 Token：
- `BOS`：前置于用户历史序列开头
- `SEP`：插入不同特征组之间

最终输入序列结构：
$$\mathbf{S} = [\texttt{BOS} ; \text{Tokenize}(\mathcal{H}) ; \texttt{SEP} ; \text{Tokenize}(\mathcal{U}) ; \texttt{SEP} ; \text{Tokenize}(\mathcal{C})]$$

**双重作用**：
1. 提供位置信息，帮助模型识别序列边界
2. 作为 **Attention Sink**，吸收多余的注意力分数，防止模型过度关注无关 Token

效果：CTR-AUC **+0.33pt**（单项最大增益之一）

#### (4) 局部注意力（Local Attention）

对历史行为序列中的大多数 Token 采用局部注意力（窗口大小），仅对候选 Token 及其附近少量 Token 保留标准因果注意力。

- 将时间复杂度从 $\mathcal{O}(L^2)$ 降至 $\mathcal{O}(L)$
- 最优窗口大小：**256**（甚至优于全局因果注意力，可能因为适当约束的局部窗口过滤了远距离"噪声"）

| 窗口大小 | CTR-AUC 提升 |
|---------|-------------|
| 64 | 较低 |
| 128 | 中等 |
| **256** | **最优** |
| INF（全局） | 次优 |

#### (5) Query Pruning（查询剪枝）

在每一层逐步剪除距离候选 Token 较远的 Query，使序列长度随层数增加而递减。最终层保留最多 128 个非候选 Token。

**效果**：
- 计算量几乎减半
- 意外地带来更好的模型性能（+0.26pt CTR-AUC）
- 推测原因：推荐系统中时序衰减的归纳偏置

数学描述（第 $l$ 层）：

$$\mathbf{Q}_{i} = \text{RMSNorm}(P(\mathbf{X}, L^{(l)})\mathbf{W}^Q_{i})$$
$$\mathbf{K}_{i} = \text{RMSNorm}(\mathbf{X}\mathbf{W}^K_{i}), \quad \mathbf{V}_{i} = \mathbf{X}\mathbf{W}^V_{i}$$
$$\mathbf{G}_{i} = \sigma(P(\mathbf{X}, L^{(l)})\mathbf{W}^G_{i})$$
$$\text{Head}_i = \mathbf{G}_{i} \odot \left( \text{Softmax}\left( \frac{\mathcal{R}(\mathbf{Q}_{i}, \mathbf{K}_{i}) + \mathbf{M}}{\sqrt{d_k}} \right) \mathbf{V}_{i}\right)$$

其中 $P(\mathbf{X}, n)$ 提取矩阵 $\mathbf{X}$ 的最后 $n$ 行，$\mathcal{R}(\cdot)$ 为 RoPE 相对位置编码，$\mathbf{G}_i$ 为注意力门控。

#### (6) QKNorm

对 Query 和 Key 施加 RMSNorm，稳定训练过程，减少注意力权重在相邻相对位置间的波动。效果：CTR-AUC **+0.12pt**。

#### (7) MoE（混合专家）FFN

采用 **DeepSeek-style MoE**（优于 Switch-style MoE，且无需调节辅助负载均衡损失超参数）：
- 最优稀疏比：**1/8**（每个 Token 激活专家数/总专家数）
- 显著扩大模型参数量（85M vs 18M），但不增加计算量（FLOPs 不变）

| 稀疏比 | 效果 |
|--------|------|
| 1（Dense FFN） | 基线 |
| 1/2 | 提升有限 |
| 1/4 | 中等提升 |
| **1/8** | **最优** |

## 三、架构其他细节

- **基础架构**：基于 LLaMA 最佳实践，采用 Pre-Norm（RMSNorm）、RoPE 相对位置编码、Swish-GLU 激活
- **Tokenization**：每个物品（历史序列 + 候选）生成一个 Token；用户画像映射为多个 Token；通过 Embedding Lookup + 拼接 + 线性投影完成
- **候选物品处理**：对角掩码 + 相同位置 ID，保证每个候选物品独立处理
- **排序头**：对候选 Token 的输出隐藏状态施加 FFN（ReLU 隐层 + Sigmoid 输出），多目标（点击/加购/购买）二元交叉熵损失加权求和
- **训练配置**：AdamW（β₁=0.9, β₂=0.99, weight_decay=0.01），BF16 混合精度，batch size=12K，默认序列长度 1K，单 epoch

## 四、基础设施

### 训练系统

基于 RecIS 框架 + Megablocks（MoE 实现），通过以下优化将 **MFU 从 13% 提升至 22%**：

- **稀疏模块**：多进程组通信（MPGC），动态调度多特征 Embedding 通信，隐藏通信延迟
- **密集模块**：自定义稀疏注意力算子（分块计算 + Mask 预加载，跳过全 Mask 块），混合精度 + 梯度累积

### 推理系统

通过 `torch.export` + AOTInductor 将动态图转为静态图，关键优化：

| 优化项 | 单节点吞吐提升 | 推理延迟降低 |
|--------|-------------|------------|
| 稀疏注意力优化 | +16.7% | -24.4% |
| 算子融合 | +10.9% | -6.3% |
| 通用优化（半精度、KV Cache、多流执行） | - | - |
| **综合** | **+29.4%** | **-29.3%** |

### Scaling 实验

在数据量、模型规模、序列长度三个维度上均验证了 SORT 的良好可扩展性：
- **数据扩展**（多 epoch / 多场景）：效果最显著，斜率最陡
- **序列长度扩展**（256 → 4K）：稳定提升
- **模型规模扩展**（Small → Large）：稳定提升

生产建议：在固定计算预算下，优先扩展数据 > 序列长度 > 模型规模。

## 六、核心贡献总结

1. **Request-Centric 样本组织**：消除冗余计算，同时支持多候选物品训练
2. **生成式预训练（GPSD 范式）**：冻结预训练 Embedding，解决过拟合，支持多 epoch 训练
3. **特殊 Token（BOS/SEP）**：作为 Attention Sink，提供边界信息，+0.33pt CTR-AUC
4. **局部注意力（窗口=256）**：将复杂度从 O(L²) 降至 O(L)，支持超长序列
5. **Query Pruning**：逐层剪枝，计算量减半，同时提升性能 +0.26pt CTR-AUC
6. **QKNorm**：稳定训练，减少注意力权重波动
7. **DeepSeek-style MoE（稀疏比 1/8）**：大幅扩展参数量（18M→85M），不增加 FLOPs
8. **系统优化**：MFU 提升至 22%，推理延迟 -44.67%，吞吐 +121.33%

## 七、相关工作对比

| 方法 | 核心思路 | 与 SORT 区别 |
|------|---------|------------|
| HSTU | 将排序视为 next-token 预测，统一序列输入 | SORT 在此基础上增加了多项优化 |
| OneTrans | 独立参数建模异构特征（规则路由 MoE） | SORT 使用数据驱动路由 MoE |
| Wukong / RankMixer | 聚焦异构特征交互 | SORT 更靠近 LLM 侧 |
| GPSD | 生成式预训练 + 稀疏 Embedding 冻结 | SORT 集成了 GPSD 并扩展了多项优化 |
